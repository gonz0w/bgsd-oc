# Roadmap: bGSD Plugin

## Milestones

- ✅ **v16.0 Enterprise Developer Team** - Phases 144-148 (shipped 2026-03-28)
- 🚧 **v16.1 Workflow Reliability & Acceleration** - Phases 149-154 (active)

## Overview

v16.1 focuses only on the two approved follow-up tracks from the milestone seed PRDs: TDD reliability and workflow acceleration. The milestone first makes TDD selection and contracts deterministic, then hardens RED/GREEN/REFACTOR validation with end-to-end proof, then removes common orchestration overhead with shared snapshots and batched updates, and finally enables fresh-context chained delivery through durable handoff artifacts and cached plan indexing. Milestone audit follow-up phases now close the remaining production wiring gaps for durable handoff persistence, stale-source enforcement, and end-to-end fresh-context proof delivery. Dynamic model configuration remains explicitly deferred beyond this milestone.

## Phase Details

### Phase 149: TDD Selection & Contract Alignment
**Goal**: Users get deterministic TDD selection decisions, visible rationale, and one canonical TDD contract before execution begins
**Depends on**: Phase 148
**Requirements**: TDD-01, TDD-02, TDD-03, TDD-04
**Success Criteria** (what must be TRUE):
  1. Planning a testable implementation phase always records whether TDD was selected or intentionally skipped, even when the roadmap provides no `**TDD:**` hint
  2. Each implementation plan includes a short human-readable rationale explaining the TDD decision
  3. TDD compliance checks report blocker, warning, or informational behavior that matches `required`, `recommended`, or omitted TDD hints
  4. Docs, workflows, skills, and CLI output all describe the same TDD terms, command names, and expected artifacts without conflicting guidance
**Plans**: 3/3 plans complete

### Phase 150: TDD Execution Semantics & Proof
**Goal**: Users can trust that `type: tdd` plans enforce real RED/GREEN/REFACTOR behavior and prove it through fixture-backed end-to-end coverage
**Depends on**: Phase 149
**Requirements**: TDD-05, TDD-06
**Success Criteria** (what must be TRUE):
  1. Running `execute:tdd` rejects a RED step unless the intended target test fails for the expected missing behavior
  2. GREEN only succeeds when the targeted behavior now passes with the minimal implementation in place
  3. REFACTOR verifies behavior remains correct after cleanup and rejects regressions that would silently break the TDD cycle
  4. Fixture-backed integration coverage proves a real `type: tdd` plan executes end to end and leaves the expected commit and summary audit trail
**Plans**: 2/2 plans complete

### Phase 151: Snapshot & Fast-Path Workflow Acceleration
**Goal**: Users can move through common phase work faster because repeated discovery and fragmented state updates are collapsed into compact, reusable commands
**Depends on**: Phase 150
**Requirements**: FLOW-01, FLOW-02, FLOW-03, FLOW-04, FLOW-05
**Success Criteria** (what must be TRUE):
  1. A single `phase:snapshot <phase>` command returns compact phase metadata, requirement IDs, artifact paths, plan index, and related execution context without repeated discovery calls
  2. Completing a plan can advance position, update progress, record metrics and decisions, and write continuity data through one batched command
  3. High-traffic init flows reuse cached phase-tree and artifact discovery data instead of rescanning directories repeatedly
  4. `discuss-phase --fast` batches low-risk clarification turns while preserving locked decisions, agent discretion, and deferred ideas
  5. `verify-work --batch N` groups routine verification checks while keeping one-at-a-time verification as the default path
**Plans**: 4/4 plans complete

### Phase 152: Cached Handoffs & Fresh-Context Delivery
**Goal**: Users can chain the standard phase-delivery loop through fresh context windows using cached plan reads and durable handoff artifacts
**Depends on**: Phase 151
**Requirements**: FLOW-06, FLOW-07, FLOW-08
**Success Criteria** (what must be TRUE):
  1. Plan indexing uses cached parsed plan data instead of reparsing markdown plans on every invocation
  2. A deterministic phase handoff JSON artifact can be created, updated, and reused to preserve compact cross-step state
  3. Users can run a chained phase-delivery flow that executes discuss, research, plan, execute, and verify through resumable fresh-context handoffs instead of one long-lived chat transcript
  4. Existing standalone commands continue to work and degrade gracefully when no handoff artifact exists
**Plans**: 5/5 plans complete

### Phase 153: Production Handoff Persistence & Resume Freshness
**Goal**: Users get durable handoff artifacts and real stale-source protection from the production workflow chain, not only from contract-level validation
**Depends on**: Phase 152
**Requirements**: FLOW-07
**Gap Closure**: Closes milestone audit wiring and flow gaps for production handoff persistence and resume freshness enforcement
**Success Criteria** (what must be TRUE):
  1. The standard discuss -> research -> plan -> execute workflow path writes or refreshes deterministic handoff artifacts as part of production execution rather than relying on tests or manual setup
  2. Resume entrypoints validate against an expected source fingerprint so stale-source blocking is enforced where users actually re-enter the chain
  3. Corrupt or partially stale handoff sets fail closed with repair guidance while still preserving latest-valid inspection and standalone fallback behavior
**Plans**: 3/3 plans complete

### Phase 154: End-to-End Fresh-Context Proof Delivery
**Goal**: Users can complete the full fresh-context delivery chain with TDD proof artifacts preserved and rendered end to end
**Depends on**: Phase 153
**Requirements**: TDD-06, FLOW-08
**Gap Closure**: Closes milestone audit proof-persistence and resumable-chain composition gaps
**Success Criteria** (what must be TRUE):
  1. TDD proof artifacts such as `TDD-AUDIT.json` persist across resumable handoff transitions and remain available to downstream summary and verification steps
  2. The discuss -> research -> plan -> execute -> verify chain is self-producing and self-preserving in production wiring rather than only documented or partially tested in isolation
  3. An integrated regression proves fresh-context resume, durable handoff continuity, and TDD proof rendering together in one end-to-end flow
**Plans**: 2/2 plans complete
