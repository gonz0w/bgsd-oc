    title: "storeSessionBundleBatch never called"

      The batch transaction API for non-sacred session state is dead code.
    remediation: |
      Identify where batch operations should occur (likely after parallel wave completion
      or at phase boundary) and wire storeSessionBundleBatch calls there.
    req_ids: [STATE-02]

  - id: GAP-002
    severity: blocker
    title: "canBatch guard never used"

      vs storeSessionBundleBatch.
    req_ids: [STATE-03]

  - id: GAP-003
    severity: blocker
    title: "Kahn sort (resolvePhaseDependencies) never triggered"
    affected_phases: [202]
    affected_requirements: [PARALLEL-02]
    evidence: |
      decision-rules.js:1163 defines resolvePhaseDependencies.
      Registered in DECISION_REGISTRY at decision-rules.js:945-953.

      Wave computation instead uses buildPhaseSnapshotInternal which parses frontmatter wave field.
    remediation: |
      Either wire enrichment.phases from roadmap phase data before evaluateDecisions,
      OR call resolvePhaseDependencies directly from execute-phase workflow.
    req_ids: [PARALLEL-02]

  - id: GAP-004
    severity: blocker

    remediation: |

## Verification Summary

**Phase 201:** All artifacts verified in isolation. Telemetry hooks exist but not wired to parallel execution.

**Phase 202:** Kahn sort and mutex primitives verified in isolation. Both are dead code — never called during execution.

**Phase 203:** Batch transaction API verified in isolation. storeSessionBundleBatch never called — batch path is dead code.

**Overall:** The milestone delivered 16 complete requirements, but 5 are affected by cross-phase wiring gaps where the Phase 202/203 artifacts exist but are not integrated into execution flows.

**Recommendation:** Phase 204 should focus on wiring the dead code paths (GAP-001 through GAP-005) rather than new feature development.
