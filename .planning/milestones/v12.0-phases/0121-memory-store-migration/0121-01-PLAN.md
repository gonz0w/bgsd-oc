**In src/lib/db.js**, append MIGRATIONS[2] to the MIGRATIONS array (after the existing migration_v2 function). The new migration creates four tables with appropriate indexes:

```sql
-- Decisions
CREATE TABLE IF NOT EXISTS memory_decisions (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  cwd       TEXT NOT NULL,
  summary   TEXT,
  phase     TEXT,
  timestamp TEXT,
  data_json TEXT
);
CREATE INDEX IF NOT EXISTS idx_mem_decisions_cwd ON memory_decisions(cwd);
CREATE INDEX IF NOT EXISTS idx_mem_decisions_phase ON memory_decisions(phase);

-- Lessons
CREATE TABLE IF NOT EXISTS memory_lessons (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  cwd       TEXT NOT NULL,
  summary   TEXT,
  phase     TEXT,
  timestamp TEXT,
  data_json TEXT
);
CREATE INDEX IF NOT EXISTS idx_mem_lessons_cwd ON memory_lessons(cwd);
CREATE INDEX IF NOT EXISTS idx_mem_lessons_phase ON memory_lessons(phase);

-- Trajectories
CREATE TABLE IF NOT EXISTS memory_trajectories (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  cwd             TEXT NOT NULL,
  entry_id        TEXT,
  category        TEXT,
  text            TEXT,
  phase           TEXT,
  scope           TEXT,
  checkpoint_name TEXT,
  attempt         INTEGER,
  confidence      TEXT,
  timestamp       TEXT,
  tags_json       TEXT,
  data_json       TEXT
);
CREATE INDEX IF NOT EXISTS idx_mem_trajectories_cwd ON memory_trajectories(cwd);
CREATE INDEX IF NOT EXISTS idx_mem_trajectories_category ON memory_trajectories(category);
CREATE INDEX IF NOT EXISTS idx_mem_trajectories_phase ON memory_trajectories(phase);

-- Bookmarks
CREATE TABLE IF NOT EXISTS memory_bookmarks (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  cwd       TEXT NOT NULL,
  phase     TEXT,
  plan      TEXT,
  task      INTEGER,
  total_tasks INTEGER,
  git_head  TEXT,
  timestamp TEXT,
  data_json TEXT
);
CREATE INDEX IF NOT EXISTS idx_mem_bookmarks_cwd ON memory_bookmarks(cwd);
```

Key design notes:
- `data_json` stores the full original JSON entry for any fields not broken out as columns — enables lossless round-tripping
- Searchable fields (summary, text, phase, category, scope) are broken out as indexed columns for LIKE queries
- `entry_id` on trajectories maps to the `id` field from trajectory.json (e.g., "tj-abc123")
- Use `CREATE TABLE IF NOT EXISTS` for idempotency

<task type="auto">
  <name>Task 2: Add PlanningCache memory migration and query methods</name>
  <files>src/lib/planning-cache.js</files>
  <action>
Add the following methods to the PlanningCache class:

**1. `migrateMemoryStores(cwd)`** — One-time JSON → SQLite migration:
- Check if memory_decisions table is empty for this cwd (SELECT COUNT(*) ... WHERE cwd = ?)
- If empty, read each JSON file from `.planning/memory/`: decisions.json, lessons.json, trajectory.json, bookmarks.json
- Parse each JSON array and INSERT all entries into corresponding SQLite tables
- For each entry, extract searchable columns AND store full JSON in `data_json`
- Run inside a single transaction per store for atomicity
- On any parse/read error for a single file, skip that file (don't abort entire migration)
- Return `{ migrated: { decisions: N, lessons: N, trajectories: N, bookmarks: N }, skipped: [] }`
- This method is idempotent — if tables already have data for this cwd, skip

**2. `searchMemory(cwd, store, query, options)`** — LIKE-based SQL search:
- `store`: 'decisions' | 'lessons' | 'trajectories' | 'bookmarks'
- `query`: text to search for (used with `%query%` LIKE pattern)
- `options`: `{ phase, category, limit, offset }`
- Build SELECT with WHERE clauses: `cwd = ?` AND `(summary LIKE ? OR data_json LIKE ?)` for decisions/lessons, `(text LIKE ? OR data_json LIKE ?)` for trajectories
- Add optional `AND phase = ?`, `AND category = ?` filters
- ORDER BY timestamp DESC (most recent first) for decisions/lessons/bookmarks, by id DESC for trajectories
- Apply LIMIT/OFFSET
- Return `{ entries: [...parsedRows], total: N }` where each entry is the parsed data_json with columns overlaid
- On Map backend, return `null` (caller falls back to JSON)

**3. `writeMemoryEntry(cwd, store, entry)`** — Single entry insert (for dual-write):
- Insert one entry into the appropriate memory_* table
- Extract searchable columns from entry, store full entry as data_json
- Return `{ inserted: true }` or `null` on Map backend
- Does NOT touch JSON files (that's the caller's responsibility)

**4. `clearMemoryStore(cwd, store)`** — Delete all entries for a store/cwd:
- DELETE FROM memory_{store} WHERE cwd = ?
- Used for re-migration or test cleanup

**5. `getBookmarkTop(cwd)`** — Get most recent bookmark:
- SELECT * FROM memory_bookmarks WHERE cwd = ? ORDER BY id DESC LIMIT 1
- Return parsed entry or null

Store-to-table mapping:
- 'decisions' → memory_decisions (searchable: summary)
- 'lessons' → memory_lessons (searchable: summary)
- 'trajectories' → memory_trajectories (searchable: text, category, scope, checkpoint_name)
- 'bookmarks' → memory_bookmarks (searchable: phase, plan)

All methods must check `this._isMap()` first and return null/no-op for Map backend.
