<objective>
Wire all state mutation commands to write SQLite-first and regenerate STATE.md as a view.

Purpose: This is the core architectural change — making SQLite the source of truth for session state. Every state mutation flows through PlanningCache session methods first, then STATE.md is regenerated from SQLite data. Manual STATE.md edits are detected and re-imported. The result: STATE.md and SQLite are always consistent, with SQLite as the programmatic interface.

Output: SQL-first state commands in state.js, a generateStateMd() function that renders STATE.md from SQLite, and SQLite-first bgsd-progress mutations.
</objective>

  <name>Task 1: Add generateStateMd() and SQL-first writes to state.js</name>
  <files>src/commands/state.js</files>
  <action>
Add SQL-first write logic to src/commands/state.js. The approach follows the established dual-write pattern: write to SQLite first, then regenerate STATE.md from SQLite data. On Map backend, fall back to existing regex-based manipulation.

**1. Add imports and helpers:**
```javascript
const { getDb } = require('../lib/db');
const { PlanningCache } = require('../lib/planning-cache');
```

Add a helper to get cache instance:
```javascript
function _getCache(cwd) {
  const db = getDb(cwd);
  return new PlanningCache(db);
}
```

**2. Add mtime-based change detection for manual STATE.md edits:**

Add `_checkAndReimportState(cwd, cache)` function that:
- Checks if STATE.md mtime has changed since last SQLite write (use file_cache table mtime for `.planning/STATE.md`)
- If stale (STATE.md newer than last known mtime), re-parse STATE.md and call `migrateStateFromMarkdown(cwd, parsedContent)` to re-import
- Update the mtime in file_cache after re-import
- This ensures manual edits to STATE.md flow back into SQLite (per CONTEXT.md: "STATE.md wins on conflict")

**3. Add `generateStateMd(cwd, cache)` function** that reads all session tables from SQLite and renders a complete STATE.md string. Format:

```markdown
# Project State

## Project Reference

See: `.planning/PROJECT.md` (updated {date})

**Core value:** {from project or data_json}
**Current focus:** Phase {phase_number} — {phase_name}

## Current Position

**Phase:** {phase_number} of {total_phases} ({phase_name}) — {status}
**Current Plan:** {current_plan}
**Status:** {status}
**Last Activity:** {last_activity}

Progress: [{progress_bar}] {progress}%

## Performance Metrics

**Velocity:**
- Total plans completed: {count} ({milestone} Phase {last_phase} Plan {last_plan})
- Average duration: ~{avg}min/plan
- Total execution time: ~{total} hours

**Recent Trend:**
{last 7-10 metric entries as "- {milestone} Phase {phase} Plan {plan}: {duration}, {tasks} tasks, {files} files ({test_count} tests)"}
- Trend: {trend}

*Updated after each plan completion*

## Accumulated Context

### Decisions

{decisions list, each as "- [{milestone} Phase {phase}]: {summary}{rationale ? ' — ' + rationale : ''}"}

### Roadmap Evolution

{from data_json if available, otherwise omit}

### Pending Todos

{todos with status='pending', each as "- {text}", or "None." if empty}

### Blockers/Concerns

{blockers with status='open', each as "- {text}", or "None." if empty}

## Session Continuity

**Last session:** {last_session}
**Stopped at:** {stopped_at}
**Next step:** {next_step}
```

The function should gracefully handle missing data (e.g., no metrics yet, no decisions). Use the current STATE.md format from the actual file as the template — preserve exact formatting.

**4. Modify each cmdState* function for SQL-first writes:**

For each mutation function, add a SQLite-first path:
- Call `_checkAndReimportState(cwd, cache)` at the start
- Write to SQLite via PlanningCache session methods
- Call `generateStateMd(cwd, cache)` to regenerate STATE.md
- Write the regenerated content to disk
- Fall back to existing regex-based approach when `cache._isMap()` returns true (or db.backend === 'map')

Specific mutations:
- `cmdStateUpdate(cwd, field, value)` → Update field in session_state via `storeSessionState(cwd, updatedState)`, regenerate
- `cmdStatePatch(cwd, patches)` → Update multiple fields in session_state, regenerate
- `cmdStateAdvancePlan(cwd)` → Update current_plan in session_state, regenerate
- `cmdStateRecordMetric(cwd, options)` → `writeSessionMetric(cwd, metric)`, regenerate
- `cmdStateUpdateProgress(cwd)` → Update progress in session_state, regenerate
- `cmdStateAddDecision(cwd, options)` → `writeSessionDecision(cwd, decision)`, regenerate
- `cmdStateAddBlocker(cwd, text)` → `writeSessionBlocker(cwd, blocker)`, regenerate
- `cmdStateResolveBlocker(cwd, text)` → Find blocker by text match, `resolveSessionBlocker(cwd, id, text)`, regenerate
- `cmdStateRecordSession(cwd, options)` → `recordSessionContinuity(cwd, continuity)`, regenerate

**5. Auto-migration on first access:**

In `cmdStateLoad` (the first command typically run), add auto-migration:
- Check if session_state row exists for cwd
- If not, parse current STATE.md and call `migrateStateFromMarkdown(cwd, content)` to populate SQLite from existing data
- This provides seamless migration — existing STATE.md files just work

**Important:** The regex-based fallback path MUST remain functional. Wrap SQLite paths in try/catch and fall through to existing regex code on any error. Pattern: "SQL-first with regex fallback."

```javascript
import { getDb, PlanningCache } from '../lib/db-cache.js';
```

**2. Add SQLite-first path in execute():**

Before the existing switch block, attempt a SQLite-first path:
- Get cache: `const db = getDb(projectDir); const cache = new PlanningCache(db);`
- Check if SQLite is available: `const hasSql = db.backend === 'sqlite';`

If `hasSql`, handle each action via PlanningCache:

- `complete-task`: Read `getSessionState(cwd)`, increment progress, `storeSessionState(cwd, updated)`, update last_activity
- `uncomplete-task`: Read `getSessionState(cwd)`, decrement progress, `storeSessionState(cwd, updated)`
- `add-blocker`: `writeSessionBlocker(cwd, { text: args.value, status: 'open', created_at: now })`
- `remove-blocker`: Query `getSessionBlockers(cwd, {status:'open'})`, find by 1-based index, `resolveSessionBlocker(cwd, id, 'Removed')`
- `record-decision`: `writeSessionDecision(cwd, { phase: state.phase, summary: args.value, timestamp: now })`
- `advance`: Read `getSessionState(cwd)`, increment plan number, `storeSessionState(cwd, updated)`

After SQLite write, call a STATE.md regeneration step. Import or inline a simplified regeneration that reads all session state from PlanningCache and writes STATE.md.

**Note:** Since bgsd-progress.js is ESM (plugin side), it uses db-cache.js's PlanningCache. The session state methods from Plan 01 must be available in the ESM PlanningCache.

**3. Preserve the existing regex fallback:**

If SQLite write fails or hasSql is false, fall through to the existing switch block with regex-based STATE.md manipulation. This ensures Map fallback works.

**4. STATE.md regeneration from plugin side:**

Since the plugin can't `require()` state.js's generateStateMd, create a lightweight `_regenerateStateMd(projectDir, cache)` helper in bgsd-progress.js (or import from a shared location). This function:
- Reads all session data from PlanningCache
- Renders STATE.md content
- Writes to disk
- Invalidates parser caches

Alternatively, shell out to `bgsd-tools.cjs state:regenerate` if such a command exists (it will after Task 1 adds generateStateMd). The simplest approach: add a `cmdStateRegenerate(cwd)` export to state.js that can be called from a new CLI subcommand `state:regenerate`, then bgsd-progress.js can exec it via child_process. However, per the architecture (no subprocess calls from plugin), the preferred approach is to inline the STATE.md template rendering in bgsd-progress.js or use a shared ESM module.

**Recommended approach:** Keep it simple — use inline STATE.md rendering in bgsd-progress.js since the format is well-defined and the function only needs to render ~50 lines of markdown. Reuse the same template as generateStateMd() in state.js.
