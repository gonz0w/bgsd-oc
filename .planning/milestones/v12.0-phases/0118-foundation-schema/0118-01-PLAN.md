<task type="auto">
  <name>Task 1: Create src/lib/db.js with dual-backend database abstraction</name>
  <files>src/lib/db.js</files>
  <action>
Create a new module `src/lib/db.js` with the following components:

**1. Feature detection function `hasSQLiteSupport()`:**
- Fast path: parse `process.version` ÔÇö if major < 22 or (major === 22 and minor < 5), return false
- Verification: `try { require('node:sqlite'); return true; } catch { return false; }`
- This dual approach follows the existing pattern in `src/lib/cache.js` for consistency

**2. `SQLiteDatabase` class (Node 22.5+ backend):**
- Constructor takes `dbPath` (string) ÔÇö the path to `.planning/.cache.db`
- Open database: `const { DatabaseSync } = require('node:sqlite'); this._db = new DatabaseSync(dbPath)`
- Set busy_timeout: Try constructor `{ timeout: 5000 }` first. If that throws (Node 22.5-22.15 lacks the option), catch and use `this._db.exec('PRAGMA busy_timeout = 5000')` after construction
- Set WAL mode: `this._db.exec('PRAGMA journal_mode = WAL')`
- Run migrations via `_runMigrations()` in constructor
- Track `this._backend = 'sqlite'` and `this._dbPath = dbPath`
- Track `this._notices = []` for status notifications

Public interface:
- `get backend()` Ôćĺ `'sqlite'`
- `get dbPath()` Ôćĺ string
- `get notices()` Ôćĺ array of notice strings (drained on read)
- `getSchemaVersion()` Ôćĺ reads PRAGMA user_version, returns number
- `exec(sql)` Ôćĺ `this._db.exec(sql)`
- `prepare(sql)` Ôćĺ `this._db.prepare(sql)`
- `close()` Ôćĺ `this._db.close()`

**3. `MapDatabase` class (fallback backend):**
- Constructor takes no args
- Stores `this._stores = new Map()` for simulating tables
- Stores `this._version = 0` (simulated schema version)
- Stores `this._backend = 'map'` and `this._dbPath = null`
- Stores `this._notices = []`
- Add notice: "Using in-memory cache (SQLite unavailable)"

Public interface (mirrors SQLiteDatabase exactly):
- `get backend()` Ôćĺ `'map'`
- `get dbPath()` Ôćĺ `null`
- `get notices()` Ôćĺ array of notice strings (drained on read)
- `getSchemaVersion()` Ôćĺ returns `this._version`
- `exec(sql)` Ôćĺ no-op (Map doesn't execute SQL)
- `prepare(sql)` Ôćĺ returns a stub object with `.get()` Ôćĺ undefined, `.all()` Ôćĺ [], `.run()` Ôćĺ `{ changes: 0 }`
- `close()` Ôćĺ clears all stores

**4. `MIGRATIONS` array:**
- Array of migration functions, indexed by version number (0-based: MIGRATIONS[0] runs to upgrade from version 0 to version 1)
- Version 1 migration (Phase 118 foundation): Create a `_meta` table with `(key TEXT PRIMARY KEY, value TEXT)` ÔÇö minimal schema to verify the migration system works. Insert a row: `key='created_at', value=new Date().toISOString()`
- Future phases (119+) will append migrations to this array

**5. `_runMigrations(db)` function (private):**
- Read current version: `const { user_version } = db.prepare('PRAGMA user_version').get()`
- If `user_version >= MIGRATIONS.length`, return (up to date)
- For each migration from `user_version` to `MIGRATIONS.length - 1`:
  - `db.exec('BEGIN')`
  - Call `MIGRATIONS[i](db._db)` (pass the raw DatabaseSync instance)
  - `db.exec('PRAGMA user_version = ' + (i + 1))`
  - `db.exec('COMMIT')`
  - On catch: `db.exec('ROLLBACK')`, then call `_deleteAndRebuild(db)`
- Note: Per research, PRAGMA user_version inside transactions needs runtime verification. If it fails, move the PRAGMA set to after COMMIT.

**6. `_deleteAndRebuild(db)` function (private):**
- Close the database connection
- Delete the db file: `fs.unlinkSync(dbPath)`
- Also delete `-wal` and `-shm` files if they exist
- Re-open and re-run all migrations from scratch
- If rebuild also fails, silently return a MapDatabase instead (it's just a cache)

**7. `getDb(cwd)` factory function (exported):**
- `cwd = cwd || process.cwd()`
- Check instance cache: `const _instances = new Map()` (module-level), keyed by resolved cwd
- If cached, return cached instance
- Check `.planning/` exists: `if (!fs.existsSync(path.join(cwd, '.planning')))` Ôćĺ return MapDatabase (do NOT create `.planning/`)
- If `hasSQLiteSupport()`:
  - Create `new SQLiteDatabase(path.join(cwd, '.planning', '.cache.db'))`
  - If the file was just created (didn't exist before), add notice: "Initialized bGSD cache"
  - If the file existed but schema version was bumped, add notice: "Upgraded to SQLite backend" (only if previous was Map)
- Else: Create `new MapDatabase()`
- Cache in `_instances` and return

**8. `closeAll()` function (exported):**
- Iterate `_instances`, call `.close()` on each, clear the map
- Useful for tests and cleanup

**Important constraints:**
- This is a CJS module (will be bundled into `bgsd-tools.cjs`)
- Zero external dependencies ÔÇö only `node:sqlite` (built-in), `fs`, `path`
- Do NOT import from other src/ modules (db.js must be self-contained for this phase)

  <name>Task 2: Implement migration failure recovery and concurrency safety</name>
  <files>src/lib/db.js</files>
  <action>
Harden the database module with failure recovery and concurrency patterns:

**1. Migration failure Ôćĺ delete-and-rebuild:**
- In `_runMigrations()`, wrap the entire migration loop in try/catch
- On any migration error: close db, delete `.cache.db` + `-wal` + `-shm` files, re-open, re-run all migrations from version 0
- If rebuild also fails: log nothing (per CONTEXT.md: "failed rebuild is silent"), return and let the instance degrade gracefully
- The db instance should set `this._degraded = true` if rebuild fails, and `getDb()` should swap to MapDatabase in that case

**2. Verify PRAGMA user_version inside transactions:**
- Test at runtime whether `PRAGMA user_version = N` works inside an explicit BEGIN/COMMIT block
- If it doesn't work (throws), implement the alternative: run COMMIT first, then set PRAGMA user_version, with a compensating mechanism if the PRAGMA set fails after commit

**3. Concurrency safety verification:**
- After setting WAL mode, verify it took effect: `const result = db.prepare('PRAGMA journal_mode').get(); if (result.journal_mode !== 'wal') { /* add notice */ }`
- After setting busy_timeout, verify: `const bt = db.prepare('PRAGMA busy_timeout').get(); /* verify it's 5000 */`

**4. DB size monitoring (soft limit with warning):**
- Add a `checkSize()` method to SQLiteDatabase
- Uses `fs.statSync(this._dbPath).size` to get file size
- If size > 50MB (configurable via `DB_SIZE_WARN_BYTES` constant), add a notice: "Cache database exceeds 50MB ÔÇö consider running cache:clear"
- Call `checkSize()` at the end of constructor (after migrations)

**5. Guard against defensive mode (Node 25.5+):**
- The `defensive` PRAGMA defaults to true in Node 25.5+
- Test that PRAGMA user_version works with defensive mode
- If it fails, explicitly set `defensive: false` in constructor options OR use `this._db.exec('PRAGMA writable_schema = ON')` before migrations
