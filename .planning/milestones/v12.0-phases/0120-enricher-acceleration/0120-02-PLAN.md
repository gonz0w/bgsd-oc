**Part A — Timing instrumentation in enricher (`src/plugin/command-enricher.js`):**

Add `performance.now()` (or `Date.now()` if performance is unavailable) timing around the enrichment:

1. Record start time at the beginning of `enrichCommand` (after the bgsd- command check).
2. Record end time just before prepending the `<bgsd-context>` XML block.
3. Add elapsed time to the enrichment object as `_enrichment_ms` (underscore prefix = internal/debug).
4. If environment variable `BGSD_DEBUG` is set or `process.env.NODE_ENV === 'development'`, log: `[bgsd-enricher] ${command} enriched in ${elapsed.toFixed(1)}ms`.
5. This timing data enables ENR-03 measurement: warm-start should show <50ms.

**Part B — Background cache warm-up (`src/plugin/index.js`):**

In the plugin's initialization hook (the `init` or `setup` function that runs when the editor loads the plugin):

1. After plugin is initialized, kick off a background warm-up by calling `getProjectState(cwd)` — this triggers parsing + SQLite write-through for all planning files, populating the cache.
2. Use `setTimeout(() => { ... }, 0)` or `queueMicrotask()` to make the warm-up non-blocking. The warm-up should not delay plugin initialization.
3. Wrap in try/catch — warm-up failure is non-fatal. Log a debug message if it fails.
4. Per CONTEXT.md decision: "Background warm on plugin load: When the plugin initializes (editor startup), kick off background cache warm-up so the first user command already has a warm cache."

**Critical constraint:** The `_enrichment_ms` field is added to the enrichment JSON but is purely informational — workflows should not depend on it. All existing enrichment fields remain unchanged.
