**Create `src/plugin/tools/bgsd-progress.js`:**
- Import `z` from `'zod'`, `getProjectState` from `'../project-state.js'`, `readFileSync`/`writeFileSync`/`mkdirSync`/`rmdirSync`/`existsSync` from `'fs'`, `join` from `'path'`, `invalidateState` from `'../parsers/state.js'`, `invalidatePlans` from `'../parsers/plan.js'`
- Export `bgsd_progress` object:
  - `description`: "Update bGSD project progress â€” mark tasks complete, add/remove blockers, record decisions, advance plan.\n\nSingle tool with an action parameter:\n- complete-task: Mark the next pending task as complete\n- uncomplete-task: Un-complete the last completed task\n- add-blocker: Add a blocker to STATE.md\n- remove-blocker: Remove a blocker by index\n- record-decision: Record a decision to STATE.md\n- advance: Advance to next plan (when current plan is complete)\n\nUpdates files on disk (STATE.md, PLAN.md). Does NOT create git commits â€” the agent handles commits separately.\n\nReturns updated state snapshot after the change."
  - `args`:
    ```
    {
      action: z.enum(['complete-task', 'uncomplete-task', 'add-blocker', 'remove-blocker', 'record-decision', 'advance']).describe('The progress action to perform'),
      value: z.string().optional().describe('Value for the action: blocker text for add-blocker, blocker index (1-based) for remove-blocker, decision text for record-decision. Not needed for complete-task, uncomplete-task, advance.'),
    }
    ```
  - `async execute(args, context)`:
    1. Get projectDir, call getProjectState â€” if null, return no_project response
    2. Validate action-specific requirements:
       - `add-blocker` and `record-decision` require `args.value`
       - `remove-blocker` requires `args.value` (blocker index)
       - Return validation_error if required value missing
    3. **File locking:** Use `mkdirSync(join(projectDir, '.planning', '.lock'))` as atomic lock. If lock exists, check staleness (stat mtime > 10s ago â†’ break it). If fresh, return runtime_error "Another operation in progress. Try again."
    4. **Action implementations** (inside lock):
       - `complete-task`: Read STATE.md, find next pending task reference, update progress. Read current PLAN.md if needed. For now, increment progress percentage (simple implementation â€” exact task tracking can be enhanced later). Update **Progress:** field and progress bar in STATE.md.
       - `uncomplete-task`: Reverse of complete-task â€” decrement progress.
       - `add-blocker`: Read STATE.md, find ### Blockers/Concerns section, append `- {value}` line. If section says "None", replace with the blocker.
       - `remove-blocker`: Read STATE.md, find ### Blockers/Concerns section, parse bullet points, remove the one at given index (1-based). If no blockers left, replace with "None".
       - `record-decision`: Read STATE.md, find ### Decisions section, append `- [Phase {currentPhase}]: {value}`.
       - `advance`: Read STATE.md, update **Current Plan:** to next plan number. If all plans complete, update **Status:** to "Complete".
    5. Write updated STATE.md back to disk
    6. **Release lock:** `rmdirSync(lockDir)`
    7. **Invalidate caches:** `invalidateState(projectDir)`, `invalidatePlans(projectDir)` if plan files were touched
    8. **Read fresh state:** Call `getProjectState(projectDir)` again to get updated snapshot
    9. Return `JSON.stringify({ success: true, action: args.action, state: { phase, plan, progress, status, blockers } })` with fresh values
    10. Use try/finally to ensure lock is always released, outer try/catch for runtime_error
