**Create `src/plugin/tools/bgsd-status.js`:**
- Import `z` from `'zod'` and `getProjectState` from `'../project-state.js'`
- Export `bgsd_status` object with:
  - `description`: Multi-line string explaining purpose, when to use, what it returns. Something like: "Get current bGSD execution state — phase, plan, tasks, progress, blockers.\n\nCall this to understand where the project is right now. Returns structured JSON with the current phase number and name, active plan, full task list with completion statuses, progress percentage, and any blockers.\n\nRequires an active bGSD project (.planning/ directory)."
  - `args`: Empty object `{}` (no parameters needed — reads current state)
  - `async execute(args, context)`: 
    1. Get projectDir from `context.directory || process.cwd()`
    2. Call `getProjectState(projectDir)` 
    3. If null, return `JSON.stringify({ status: 'no_project', message: 'No .planning/ directory found. Run /bgsd-new-project to initialize a project.' })`
    4. Extract from state: phase (number + name parsed from "73 — Name" format), currentPlan, status, progress percentage
    5. Extract blockers from state.getSection('Blockers/Concerns') — parse bullet points into array, filter out "None"
    6. Extract full task list from plans array: for each plan, for each task, include { plan: "P01", taskNumber: N, name, files, status: "pending"|"complete" }. Use plan frontmatter and task list. Since the parser doesn't track completion, all tasks are "pending" by default — bgsd_progress (Plan 02) will add completion tracking
    7. Build response object with fields always present (null for missing):
       ```
       {
         phase: { number: "74", name: "Custom LLM Tools" } | null,
         plan: { id: "P01", status: "in_progress" } | null,
         progress: 45 | null,
         tasks: [{ plan: "P01", task: 1, name: "Task 1: ...", files: [...], status: "pending" }],
         blockers: ["Bundle size concern", ...] | [],
       }
       ```
    8. Return `JSON.stringify(result)`
    9. Wrap body in try/catch — catch returns `JSON.stringify({ error: 'runtime_error', message: 'Failed to read project state: ' + err.message })`

**Create `src/plugin/tools/index.js`:**
- Import all tool definitions from individual files (bgsd-status.js for now)
- Export a `getTools(registry)` function that:
  1. Calls `registry.registerTool('status', bgsd_status)` for each tool
  2. Returns `registry.getTools()` — the plain object of all registered tools
- This is the single wiring point. Plan 02 tools will add imports here.

**Update `src/plugin/index.js`:**
- Import `getTools` from `'./tools/index.js'`
- Replace the commented-out `// tool: registry.getTools()` line with: `tool: getTools(registry),`
- This wires tools into the plugin's return object so the host editor discovers them

**Important patterns (from CONTEXT.md / RESEARCH.md):**
- Use `z` from `'zod'` directly — NOT `tool()` helper from `@opencode-ai/plugin`
- Always `return JSON.stringify(result)` — never return a plain object
- Fields always present with null for missing data — predictable shape
- Error types: validation_error (bad args) vs runtime_error (operational failures)
- Actionable error messages: "Phase 99 not found. Run /bgsd-progress to see available phases."
- No metadata (timing, cache freshness) in responses

**Create `src/plugin/tools/bgsd-plan.js`:**
- Import `z` from `'zod'`, `getProjectState` from `'../project-state.js'`, `parseRoadmap` from `'../parsers/roadmap.js'`, `parsePlans` from `'../parsers/plan.js'`
- Export `bgsd_plan` object:
  - `description`: "Get roadmap overview or detailed phase information.\n\nTwo modes:\n- No args: returns all phases with status, goal, and plan count (roadmap summary)\n- With phase number: returns detailed phase info (goal, requirements, success criteria, dependencies) plus plan contents (tasks, objectives) if plans exist\n\nUse no-args mode to understand project structure. Use phase mode to dive into specific phase details."
  - `args`: `{ phase: z.coerce.number().optional().describe('Phase number to get details for. Omit for roadmap overview.') }`
  - `async execute(args, context)`:
    1. Get projectDir from `context.directory || process.cwd()`
    2. Call `getProjectState(projectDir)` — if null, return no_project response
    3. **No-args mode** (args.phase is undefined):
       - Map over `roadmap.phases` array, for each phase return: `{ number, name, status, goal, planCount }`
       - Include current milestone info from `roadmap.currentMilestone`
       - Return `JSON.stringify({ phases: [...], currentMilestone: { name, version, status } | null })`
    4. **Phase mode** (args.phase is a number):
       - Call `roadmap.getPhase(args.phase)` — if null, return `JSON.stringify({ error: 'validation_error', message: 'Phase ${args.phase} not found in roadmap. Call bgsd_plan with no args to see available phases.' })`
       - Get plans via `parsePlans(args.phase, projectDir)`
       - Return `JSON.stringify({ phase: { number, name, goal, dependsOn, requirements, successCriteria, plans: plans.map(p => ({ plan: p.frontmatter.plan, wave: p.frontmatter.wave, objective: p.objective, tasks: p.tasks.map(t => ({ name: t.name, type: t.type, files: t.files })), requirements: p.frontmatter.requirements })) } })`
    5. Wrap in try/catch with runtime_error envelope

**Create `src/plugin/tools/bgsd-context.js`:**
- Import `z` from `'zod'`, `getProjectState` from `'../project-state.js'`
- Export `bgsd_context` object:
  - `description`: "Get task-scoped context for the current or specified task.\n\nReturns file paths, line ranges, and summaries relevant to a specific task — not actual file contents (use the Read tool for that).\n\nDefaults to the current task from STATE.md. Pass a task number to get context for a different task in the current plan."
  - `args`: `{ task: z.coerce.number().optional().describe('Task number within current plan. Defaults to current task.') }`
  - `async execute(args, context)`:
    1. Get projectDir from `context.directory || process.cwd()`
    2. Call `getProjectState(projectDir)` — if null, return no_project response
    3. Find current plan from `plans` array (match state.currentPlan)
    4. If no plans exist, return `JSON.stringify({ error: 'validation_error', message: 'No plans found for current phase. Run /bgsd-plan-phase to create plans.' })`
    5. Determine task index: `args.task ? args.task - 1 : 0` (1-indexed input, 0-indexed array)
    6. If task index out of bounds, return validation_error: "Task N not found. Current plan has M tasks."
    7. Extract from the task: name, files, action, done criteria
    8. Also include: plan objective, plan verification section, plan files_modified from frontmatter
    9. Return `JSON.stringify({ task: { number: taskIndex + 1, name, files, action, done }, plan: { id: planId, objective, verification, filesModified } })`
    10. Wrap in try/catch with runtime_error envelope

**Update `src/plugin/tools/index.js`:**
- Add imports for `bgsd_plan` and `bgsd_context`
- Register both: `registry.registerTool('plan', bgsd_plan)` and `registry.registerTool('context', bgsd_context)`

**Important patterns:**

- Use `z.coerce.number()` for phase/task args — lenient with LLM type mismatches
