Create `src/plugin/parsers/` directory with 4 parser modules + index. Each parser follows the same pattern per CONTEXT.md decisions:
- Return structured typed objects with accessor methods
- Return null when file can't be parsed (corrupt, missing, wrong format)
- Cache parsed results until explicitly invalidated via `invalidate()` call
- Read-only parsers (no write operations)
- Auto-discover files from CWD — no path argument required (optional override)
- Parsed objects are immutable (Object.freeze)
- Include raw markdown alongside structured data
- Lenient parsing — best-effort extraction, skip/null unrecognizable parts

**1. `src/plugin/parsers/state.js`** — `parseState(cwd?)`:
Extract parsing logic from `src/commands/state.js` (stateExtractField pattern) and `src/lib/helpers.js` (safeReadFile):
- Auto-discovers `.planning/STATE.md` from `cwd || process.cwd()`
- Reads file with `fs.readFileSync`
- Extracts fields using `**FieldName:** value` regex pattern (same as stateExtractField)
- Returns frozen object:
  ```
  {
    raw: string,              // Full markdown content
    phase: string | null,     // Current phase number + name
    currentPlan: string | null,
    status: string | null,
    lastActivity: string | null,
    progress: number | null,  // 0-100 from progress bar
    getField(name): string | null,
    getSection(name): string | null,
  }
  ```
- Cache: module-level Map keyed by cwd, with `invalidateState(cwd?)` export

**2. `src/plugin/parsers/roadmap.js`** — `parseRoadmap(cwd?)`:
Extract from `src/commands/roadmap.js` (cmdRoadmapGetPhase) and `src/lib/helpers.js` (getRoadmapPhaseInternal):
- Auto-discovers `.planning/ROADMAP.md` from `cwd || process.cwd()`

  - Milestone list with status (✅ complete, 🔲 pending)
  - Phase list with completion status, plan counts
  - Phase details (goal, depends_on, requirements, success_criteria)
  - Progress table
- Returns frozen object:
  ```
  {
    raw: string,
    milestones: [{ name, status, phases }],
    phases: [{ number, name, status, planCount }],
    progress: [{ phase, milestone, plansComplete, status, completed }],
    getPhase(num): { number, name, goal, dependsOn, requirements, successCriteria, plans } | null,
    getMilestone(name): { ... } | null,
    currentMilestone: { ... } | null,
  }
  ```

  - Returns frozen object or null
- `parsePlans(phaseNum, cwd?)`: Discover and parse all PLAN.md files for a phase
  - Find phase directory via `.planning/phases/{num}-*/`
  - Glob for `*-PLAN.md` files
  - Parse each with parsePlan()
  - Return frozen array of parsed plans, sorted by plan number

<task type="auto">
  <name>Task 2: Create tool registry with bgsd_ prefix enforcement and snake_case validation</name>
  <files>src/plugin/tool-registry.js</files>
  <action>
Create `src/plugin/tool-registry.js` implementing tool naming enforcement per CONTEXT.md decisions:

1. `createToolRegistry(safeHookFn)` function — factory that returns a registry instance:
   - `safeHookFn`: reference to safeHook for wrapping tool handlers

2. `registerTool(name, definition)` method:
   - `name`: tool name string
   - `definition`: `{ description, args, execute }` — includes schema/description metadata (agent's discretion decision: include full definition for Phase 74 compatibility with OpenCode's `tool()` helper)
   
   Behavior:
   a. **Auto-prefix**: If `name` doesn't start with `bgsd_`, prepend it silently
      - `"status"` → `"bgsd_status"`
      - `"bgsd_status"` → `"bgsd_status"` (no double prefix)
   b. **Enforce snake_case**: Validate normalized name matches `/^bgsd_[a-z][a-z0-9_]*$/`
      - If invalid: throw `Error("Tool name must be snake_case: ${normalized}")`
      - Examples of invalid: `bgsd_Status` (uppercase), `bgsd_my-tool` (hyphen), `bgsd_` (nothing after prefix)
   c. **Duplicate detection**: If `registry.has(normalized)`:
      - Log warning: `console.warn("[bGSD] Tool '${normalized}' already registered — overwriting")`
      - Continue with overwrite (don't error)
   d. **Wrap handler in safeHook**: `definition.execute = safeHookFn('tool:' + normalized, definition.execute)`
   e. **Store in registry**: `registry.set(normalized, { name: normalized, ...definition })`
   f. **Return normalized name** (so caller knows the final name)

3. `getTools()` method:
   - Returns a plain object mapping tool names to their full definitions
   - Used by plugin entry point to populate the `tool` return key
   - Format: `{ bgsd_status: { description, args, execute }, ... }`

4. Internal state:
   - `registry = new Map()` — tool name → definition
   - No public has/get/list API (CONTEXT.md: registry is internal only)
   - No unregistration API (permanent for session)
   - No tool count limit

5. Export: `module.exports = { createToolRegistry }`
