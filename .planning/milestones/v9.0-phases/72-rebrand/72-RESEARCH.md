# Phase 72: Rebrand - Research

**Researched:** 2026-03-08
**Domain:** Codebase-wide naming migration (gsd-* to bgsd-*)
**Confidence:** HIGH

## Summary

Phase 72 is a comprehensive rename across ~614 active-code references spanning 7 directories (src/, commands/, workflows/, templates/, agents/, skills/, and root files). The rename touches 6 distinct naming patterns: config folder (`get-shit-done` -> `bgsd-oc`), CLI binary (`gsd-tools` -> `bgsd-tools`), environment variables (`GSD_HOME/DEBUG/PROFILE` -> `__OPENCODE_CONFIG__/bgsd-oc/DEBUG/PROFILE`), agent file names (`gsd-*.md` -> `bgsd-*.md`), agent name references in code, and all usage/help strings.

Phase 71 already established the `BGSD_DEBUG` pattern in the plugin layer (`src/plugin/safe-hook.js`) and the `bgsd_` tool prefix convention, providing a migration template. The CLI source (`src/lib/output.js`, `src/lib/profiler.js`, `src/router.js`) still uses the old `GSD_DEBUG` and `GSD_PROFILE` patterns. The build pipeline (`build.cjs`) currently outputs to `bin/gsd-tools.cjs` and must be updated to `bin/bgsd-tools.cjs`. The `install.js` needs migration logic added for existing users.

**Primary recommendation:** Execute in dependency order: (1) env vars + source code first, (2) build pipeline + binary rename, (3) agent files, (4) workflows/commands/skills, (5) install.js migration + deploy.sh, (6) docs/tests/package.json, (7) validation sweep.

## Common Pitfalls

### Pitfall 1: Partial Rename Breaks Tests
**What goes wrong:** Renaming the binary in build.cjs but not in the test file, or renaming env vars in source but not in test assertions (e.g., tests still assert `[GSD_DEBUG]` prefix in output).
**Why it happens:** The test file (`bin/gsd-tools.test.cjs`) has 63+ references to old naming spread across 18,000+ lines.
**How to avoid:** Rename the test file last, after source and build changes. Use find-and-replace carefully. Run `npm test` after each wave.
**Warning signs:** Tests fail with "Cannot find module" or "command not found" errors.

### Pitfall 2: Debug Log Format is Tested
**What goes wrong:** The `debugLog()` function produces `[GSD_DEBUG] context: message` format. Tests specifically assert this prefix. Changing the env var name without changing the log prefix format breaks 4+ tests.
**Why it happens:** The output format string in `src/lib/output.js:191` uses the literal `[GSD_DEBUG]` as a prefix.
**How to avoid:** Change both the env check AND the output format string simultaneously: `process.env.BGSD_DEBUG` and `[BGSD_DEBUG]`.
**Warning signs:** Debug logging tests fail on prefix matching.

### Pitfall 3: Agent PATH SETUP Glob Pattern
**What goes wrong:** The agent PATH SETUP uses `ls -d $HOME/.config/*/get-shit-done` which globs across any intermediate directory name. Changing to `bgsd-oc` but forgetting to update the glob pattern breaks all agents.
**Why it happens:** The glob pattern is in 10 separate files, easy to miss one.
**How to avoid:** Search-and-replace across all `agents/gsd-*.md` files. Verify with `grep -c 'get-shit-done' agents/*.md` after.
**Warning signs:** Agents fail to resolve their home directory.

### Pitfall 4: Deployed Install Has Old Names
**What goes wrong:** The dev workspace gets renamed but the deployed install at `~/.config/opencode/get-shit-done/` still has old names. Running `/bgsd-*` commands after deploy uses stale code.
**Why it happens:** deploy.sh copies to the old destination path.
**How to avoid:** Update deploy.sh DEST path first. After deploying with the new path, manually clean up `~/.config/opencode/get-shit-done/` and old agent files.
**Warning signs:** Commands work in dev but not in production.

### Pitfall 5: Workflow Placeholder Paths
**What goes wrong:** Workflows use `__OPENCODE_CONFIG__/get-shit-done/bin/gsd-tools.cjs` which gets substituted at deploy time by `deploy.sh` and `install.js`. Both the placeholder path AND the substitution logic must be updated.
**Why it happens:** The placeholder is baked into the source markdown files, and substitution happens at deploy/install time.
**How to avoid:** Update all `__OPENCODE_CONFIG__/get-shit-done/` references to `__OPENCODE_CONFIG__/bgsd-oc/` in source files. Deploy.sh and install.js substitution logic stays the same (it replaces `__OPENCODE_CONFIG__` with the actual path).
**Warning signs:** Workflows fail to find gsd-tools binary after deploy.

### Pitfall 6: build.cjs Manifest Agent Filter
**What goes wrong:** `build.cjs:267` uses `name.startsWith('gsd-')` to filter agent files for the manifest. After renaming agents to `bgsd-*`, the manifest won't include any agents.
**Why it happens:** The build script explicitly filters for `gsd-` prefix.
**How to avoid:** Update the filter to `name.startsWith('bgsd-')` simultaneously with the file renames.
**Warning signs:** Deployed install has 0 agents.

### Pitfall 7: install.js Uninstall Pattern
**What goes wrong:** `install.js` uninstall removes `gsd-*.md` from agents dir. After rename, it needs to remove `bgsd-*.md`. But also needs to clean up OLD `gsd-*.md` files from previous installs.
**Why it happens:** The uninstall pattern is hardcoded to `gsd-` prefix.
**How to avoid:** Update uninstall to remove both `gsd-*.md` and `bgsd-*.md` agent files. The migration path in install should handle the old-to-new transition.
**Warning signs:** Old agent files persist after uninstall/reinstall.

### Pitfall 8: Usage Strings Are User-Facing
**What goes wrong:** The 149 usage strings in `src/lib/constants.js` show `gsd-tools` as the command name. Users see help output with the old name.
**Why it happens:** These are string literals, not derived from the binary name.
**How to avoid:** Global find-replace `gsd-tools` -> `bgsd-tools` in constants.js. The pattern is consistent: `Usage: gsd-tools <command>`.
**Warning signs:** Help output shows old binary name.

| File renames | Manual mv commands | `git mv` for tracked files | Preserves git history |

if (existsSync(OLD_DEST) && !existsSync(NEW_DEST)) {
  console.log(`  Migrating ${OLD_DEST} -> ${NEW_DEST}...`);

  rmSync(OLD_DEST, { recursive: true, force: true });

// Also migrate old agent files: gsd-*.md -> bgsd-*.md
if (existsSync(AGENT_DIR)) {

      const newName = f.replace(/^gsd-/, "bgsd-");
      cpSync(join(AGENT_DIR, f), join(AGENT_DIR, newName));
      unlinkSync(join(AGENT_DIR, f));

  }

## Risk Assessment

### Migration Risk (RBND-07)
**Risk:** MEDIUM — Migration is a file-level directory move, but edge cases include:
- Partial old installs (some files missing)
- Concurrent sessions during migration
- Old agent files in `~/.config/opencode/agents/gsd-*.md` that need cleanup
- Symlinks or custom configurations

**Mitigation:** Migration should be copy-then-delete (not rename), with existence checks at each step. Old agent files should be cleaned up as part of migration.

### Backward Compatibility Risk
**Risk:** LOW — STATE.md explicitly declares: "Rebrand is a clean break — no backward compatibility with old `gsd-*` naming." REQUIREMENTS.md explicitly marks backward compatibility as out of scope for v9.0.

### Test Breakage Risk
**Risk:** MEDIUM — 63+ test references to old naming must all be updated atomically with source changes. Tests cannot pass in a partial-rename state.

**Mitigation:** Rename source + tests in the same commit wave. Run full test suite after each wave.
