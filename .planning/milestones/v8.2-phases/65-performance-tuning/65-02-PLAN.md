**Step 1: Profile current state**

Run profiling on the three primary init commands to identify bottlenecks:

Capture timing breakdowns from profiler output. Look for:
- Which sub-operations take the most time
- Redundant file reads (same file read multiple times)
- Expensive operations that could be skipped for certain init variants

**Step 2: Apply optimizations based on profiling data**

Known optimization targets (from CONTEXT.md decisions):

1. **"Light init" paths** — Init commands that don't need AST/codebase-intel should skip loading those modules. Check which init variants call `autoTriggerCodebaseIntel()` and whether it's always needed. If `codebase_freshness.stale` is the only reason it triggers, consider deferring.

2. **Skip unnecessary aggregation** — `cmdInitPlanPhase` and `cmdInitExecutePhase` aggregate state, config, phase info, roadmap section, plan details, codebase context, environment, worktree info. Profile which of these sub-calls are slowest and whether any can be conditionally skipped.

3. **Reduce require() chain at startup** — In `router.js`, check if any top-level requires can be made lazy. The profiler and format modules are loaded on every invocation — profile whether this is significant.

4. **Avoid re-reading config** — `loadConfig()` is called from multiple entry points. Verify the config cache in `config.js` (`_configCache`) is working correctly across a single invocation.

Do NOT touch `src/lib/ast.js` or acorn — that's Plan 01's scope.

<task type="auto">
  <name>Task 2: Reduce file I/O and capture final baselines</name>
  <files>src/lib/helpers.js, src/commands/init.js</files>

**Step 1: Instrument and measure file reads**

Add temporary read counting to identify which files are read most frequently during init:
- Use `GSD_PROFILE=1` profiling to see `file:read:*` and `file:read-uncached:*` entries
- Count unique vs. duplicate file reads
- Identify files read >2 times in a single init invocation

Focus on known suspects from CONTEXT.md: "531 reads feels high. Expect redundant re-parsing of ROADMAP.md, STATE.md, and phase directories across a single command lifecycle."

**Step 2: Optimize read patterns**

1. **Ensure `cachedReadFile` is used consistently** — Grep for `fs.readFileSync` in init.js and helpers.js. Any direct reads should use `cachedReadFile` or `safeReadFile` instead (unless they need guaranteed-fresh data after a write).

2. **Avoid redundant `getPhaseTree()` calls** — This function scans the entire `.planning/phases/` directory tree. It has `_phaseTreeCache` but verify it's used in all call sites within init.js. Multiple init sub-functions might each call `findPhaseInternal()` which internally calls `getPhaseTree()`.

3. **Batch directory reads** — If init reads the same directory multiple times (e.g., listing phase plans), ensure `cachedReaddirSync` is used.

4. **Consider parsing once, sharing result** — If both `getRoadmapPhaseInternal()` and another function parse ROADMAP.md, ensure they share the cached content via `cachedReadFile` rather than each reading independently.

**Step 3: Capture final baselines**

After optimizations, run the full baseline capture:
```bash

# File I/O count
GSD_PROFILE=1 node bin/gsd-tools.cjs init plan-phase 65 2>&1 | grep -c 'file:read'
```
