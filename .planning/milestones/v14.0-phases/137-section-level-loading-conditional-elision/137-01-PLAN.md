# Plan 01: Conditional Elision Engine + Workflow Annotations

## Objective

<!-- data -->
This plan contributes to: The enricher loads only the workflow section(s) relevant to the current step instead of the full workflow, and conditional features are elided when bgsd-context decisions indicate they don't apply — delivering per-invocation context savings

This plan implements the conditional elision engine in the command enricher: parsing `<!-- section: name if="condition" -->` markers, evaluating conditions against bgsd-context decisions/enrichment fields, stripping false sections from workflow content in `output.parts`, and annotating the three target workflows (execute-plan, execute-phase, quick) with conditional markers.

### Task 2: Implement conditional section parser and elision engine

- **Type:** auto
- **Files:**
  - `src/plugin/command-enricher.js` (modify)

- **Action:**

  Create an `elideConditionalSections(text, enrichment)` function in `command-enricher.js` that:

  1. **Parses conditional section markers** using regex: `<!--\s*section:\s*(\S+)\s+if="([^"]+)"\s*-->` to extract section name and condition key. Also matches closing `<!-- /section -->` markers.
  2. **Evaluates each condition** against the enrichment object:
     - Simple field lookup: `enrichment[conditionKey]` — truthy = keep, falsy = elide
     - Decision lookup: `enrichment.decisions?.[conditionKey]?.value` — truthy = keep, falsy = elide
     - Boolean string handling: `"true"` = keep, `"false"` = elide
     - Missing key = keep (fail-open: don't elide if condition can't be evaluated)
  3. **Strips false sections**: Removes everything from opening `<!-- section: name if="..." -->` through closing `<!-- /section -->` inclusive, for sections whose condition evaluates to false.
  4. **Preserves unconditional sections**: Markers without `if=` are never touched.
  5. Returns the modified text.

  Condition evaluation order: `enrichment[key]` first (direct field), then `enrichment.decisions?.[key]?.value` (decision lookup). First match wins.

  Edge cases to handle:
  - Nested sections: not supported (per CONTEXT.md — single conditions only). If encountered, process outermost first.
  - Unclosed conditional sections: treat as extending to EOF (consistent with `extractSectionsFromFile`).
  - Multiple conditional sections in same text: process all independently.

- **Verify:** Unit tests in Task 5 cover all cases. Function is exported for testing.

- **Done:** `elideConditionalSections(text, enrichment)` correctly strips sections whose conditions are false and preserves all others.

### Task 3: Wire elision into enrichCommand() and output.parts processing

- **Type:** auto
- **Files:**
  - `src/plugin/command-enricher.js` (modify)

- **Action:**

  After the `<bgsd-context>` block is prepended to `output.parts` (line 572-576), add a post-processing step that:

  1. Iterates over `output.parts` (skipping the just-prepended bgsd-context block at index 0).
  2. For each text part that contains `<!-- section:` markers with `if=` attributes, calls `elideConditionalSections(part.text, enrichment)`.
  3. Replaces `part.text` with the elided result.
  4. Tracks elision stats: `{ sections_elided: number, tokens_saved_estimate: number }` and adds to the enrichment object as `_elision` (underscore prefix = debug field).

  **Important:** The elision must happen AFTER the bgsd-context block is prepended (so the enrichment object is complete with decisions), but within the same `enrichCommand()` call. Place the elision code between the bgsd-context prepend (line 576) and the closing brace of `enrichCommand()`.

  Also add `elision_applied: true` to the `<bgsd-context>` JSON when elision occurs, so agents know sections may have been removed. Update the bgsd-context block in `output.parts[0]` after elision to include this flag.

- **Verify:** Run a `/bgsd-execute-phase` command with `ci_enabled: false` in the project config. Confirm the `ci_quality_gate` section is absent from the workflow content the agent receives. Confirm `_elision` stats in bgsd-context.

- **Done:** The enricher strips conditional sections from workflow content in output.parts based on enrichment state. Stats are tracked in `_elision` debug field.

### Task 5: Unit tests for conditional elision

- **Type:** auto
- **Files:**
  - `tests/enricher-elision.test.cjs` (create)

- **Action:**

  Create a test file covering the `elideConditionalSections` function. Tests:

  1. **Basic elision**: Section with `if="is_tdd"` is removed when `enrichment.is_tdd` is falsy
  2. **Basic preservation**: Section with `if="is_tdd"` is kept when `enrichment.is_tdd` is truthy
  3. **Decision lookup**: Section with `if="ci_enabled"` is removed when `enrichment.ci_enabled` is false but kept when true
  4. **Fail-open**: Section with `if="unknown_key"` is preserved (missing key = keep)
  5. **Unconditional sections preserved**: `<!-- section: execute -->` without `if=` is never modified
  6. **Multiple conditional sections**: Two conditional sections in same text, one true one false — only false is removed
  7. **Mixed conditional/unconditional**: Text with both types — only conditional false sections removed
  8. **Empty enrichment**: No enrichment fields — all conditional sections preserved (fail-open)
  9. **Nested content preservation**: Content between conditional markers is fully removed (including nested HTML comments)
  10. **Integration test**: Simulate enrichCommand flow with mock output.parts containing workflow-like content, verify elision occurs

  Use the project's existing test patterns (likely `node:test` or similar). Import `elideConditionalSections` from the built module or test the function directly.
