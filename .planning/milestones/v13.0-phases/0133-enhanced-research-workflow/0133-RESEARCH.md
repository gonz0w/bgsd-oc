- **Command registration:** New subcommands slot into the existing `research` namespace in `src/router.js` (lines 1285-1306), backed by `src/commands/research.js` which already exports 8 command handlers (`cmdResearchCapabilities`, `cmdResearchYtSearch`, etc.). Every file has: `**Confidence:** HIGH|MEDIUM|LOW` on line 5, `## Sources` with `### Primary (HIGH confidence)`, `### Secondary (MEDIUM confidence)`, `### Tertiary (LOW confidence)` subsections, a `## Metadata` section with `**Confidence breakdown:**` and `**Research date:**`. The profile is generated by calling `research:score` on each RESEARCH.md file and surfacing LOW-confidence flags.

## User Constraints (from CONTEXT.md)

These decisions are LOCKED â€” planner must honor exactly:

1. **Output format:** All commands (`research:score`, `research:gaps`) output raw structured JSON to stdout â€” no human-formatted summaries, no flags like `--pretty`
2. **Cache file:** `research:score` caches the profile as a visible file `research-score.json` in the phase directory (not a dotfile)
3. **`research:gaps` is a wrapper:** Extracts `flagged_gaps[]` from the cached `research-score.json` â€” not an independent computation
4. **Legacy grade dropped:** The A-F letter grade is dropped entirely â€” clean break, no backward-compatible `overall_grade` field
5. **Confidence thresholds:** Determined by composite score â€” LOW requires multiple weak factors (few sources AND old AND no official docs), not a single red flag; hardcoded defaults, no config.json surface
6. **Profile includes computed `confidence_level`:** Three tiers: HIGH/MEDIUM/LOW â€” consumers don't recompute
7. **Conflicts surfaced as top-level array:** `conflicts[]` flat alongside `flagged_gaps[]`, each entry: `{claim, source_a, source_b}`, listed neutrally with no `suggested_source` or trust ranking
8. **Gap objects are rich:** `{ gap, severity, section, suggestion }` â€” includes description, severity (HIGH/MEDIUM/LOW), location in doc, and what to research
9. **`new-milestone.md` surfaces HIGH + MEDIUM gaps only** â€” LOW filtered out to reduce noise
10. **`research:gaps` returns ALL gaps** with no filtering â€” consumers filter if needed
11. **Conflict detection sensitivity:** Agent's discretion â€” determines what counts as a "disagreement" between sources

### Pattern 1: RESEARCH.md Parsing

The parsing approach uses regex to extract structured data from the consistent RESEARCH.md template:

1. **Header metadata:** `**Confidence:** (HIGH|MEDIUM|LOW)` on line 5, `**Researched:** YYYY-MM-DD` on line 3
2. **Sources section:** Split by `### Primary`, `### Secondary`, `### Tertiary` headings; count bullet items per tier
3. **Official docs detection:** Check if any source entry in Primary/Secondary contains URLs from known official documentation domains (docs.*, official docs references, or contains "official" in description)
4. **Confidence breakdown section:** Parse individual domain confidence levels from the `## Metadata` `**Confidence breakdown:**` section
5. **Gap detection:** Identify sections where the researcher flagged LOW confidence in specific domains, and any `## Open Questions` items
6. **Conflict detection:** Look for language patterns indicating source disagreement â€” phrases like "contradicts", "disagrees", "conflicts with", "unlike X which says", or explicit conflict notes

```javascript
// Pseudo-code for parseResearchFile(content)
function parseResearchFile(content) {
  const lines = content.split('\n');
  
  // Extract header confidence
  const confidenceMatch = content.match(/\*\*Confidence:\*\*\s*(HIGH|MEDIUM|LOW)/i);
  const headerConfidence = confidenceMatch ? confidenceMatch[1].toUpperCase() : null;
  
  // Extract research date
  const dateMatch = content.match(/\*\*Researched?:\*\*\s*(\d{4}-\d{2}-\d{2})/);
  const researchDate = dateMatch ? dateMatch[1] : null;
  
  // Count sources per tier
  const sourceSections = extractSourceSections(content);
  // { primary: [...], secondary: [...], tertiary: [...] }
  
  // Detect official docs
  const hasOfficialDocs = sourceSections.primary.some(s => isOfficialDoc(s));
  
  // Extract confidence breakdown
  const breakdownEntries = extractConfidenceBreakdown(content);
  // [{ domain: 'X', level: 'HIGH' }, ...]
  
  // Detect gaps
  const gaps = extractGaps(content, breakdownEntries);
  
  // Detect conflicts
  const conflicts = detectConflicts(content);
  
  return { headerConfidence, researchDate, sourceSections, hasOfficialDocs, breakdownEntries, gaps, conflicts };
}
```

### Pattern 2: Composite Confidence Scoring

Per CONTEXT.md: LOW requires multiple weak factors, not a single red flag. The scoring function:

```javascript
function computeConfidenceLevel(parsed) {
  let negativeSignals = 0;
  
  const totalSources = parsed.sourceSections.primary.length 
    + parsed.sourceSections.secondary.length 
    + parsed.sourceSections.tertiary.length;
  
  if (totalSources < 3) negativeSignals++;           // Few sources
  if (!parsed.hasOfficialDocs) negativeSignals++;     // No official docs
  if (parsed.researchDate) {                          // Old research
    const daysSince = daysBetween(parsed.researchDate, new Date());
    if (daysSince > 90) negativeSignals++;
  }
  if (parsed.conflicts.length > 0) negativeSignals++; // Multi-source conflicts
  
  const highConfidencePct = totalSources > 0 
    ? parsed.sourceSections.primary.length / totalSources * 100 
    : 0;
  if (highConfidencePct < 30) negativeSignals++;       // Most sources are low-confidence
  
  if (negativeSignals >= 3) return 'LOW';
  if (negativeSignals >= 1) return 'MEDIUM';
  return 'HIGH';
}
```

1. **Don't use an LLM for scoring.** The CLI is deterministic â€” no API calls. Scoring is regex + heuristic.

3. **Don't add a backward-compatible `overall_grade` field.** Clean break from A-F grading.
4. **Don't filter gaps in `research:gaps`.** It returns ALL gaps; `new-milestone.md` filters HIGH+MEDIUM.
5. **Don't add trust ranking to conflicts.** They're listed neutrally per CONTEXT.md.

## Common Pitfalls

### Pitfall 1: Inconsistent RESEARCH.md Formats

**What goes wrong:** Older RESEARCH.md files (pre-template standardization) may have slightly different heading structures or missing sections.
**Why it happens:** The template evolved over 42 files and 13 milestones. Early files may lack `## Metadata` or have different source heading styles.
**How to avoid:** Make parsing defensive â€” every regex extraction should have a fallback (null, empty array, 0). Test against at least 3 real RESEARCH.md files from different eras (v1.0, v8.0, v13.0).
**Warning signs:** `null` values in profile fields where numbers are expected; empty `flagged_gaps[]` on files that clearly have open questions.

### Pitfall 2: research-score.json Stale After Re-Research

**What goes wrong:** A user re-runs research for a phase, producing a new RESEARCH.md, but the cached `research-score.json` still reflects the old version.
**Why it happens:** `research:score` writes the cache on each invocation, but `research:gaps` reads the cache without checking freshness.
**How to avoid:** `research:score` always overwrites the cache file. `research:gaps` could optionally check mtime of RESEARCH.md vs cache, but per CONTEXT.md it just reads the cache â€” document this as expected behavior.
**Warning signs:** `research:gaps` returning gaps that don't exist in the current RESEARCH.md.

### Pitfall 3: Conflict Detection False Positives

**What goes wrong:** The conflict detector flags normal comparative language ("unlike library X, library Y does Z") as a source conflict.
**Why it happens:** Simple keyword matching on "contradicts", "disagrees", etc. without context.
**How to avoid:** Require conflicts to be near a source citation or within the Sources section. The CONTEXT.md says conflict detection sensitivity is at the agent's discretion â€” err on the side of precision over recall.
**Warning signs:** Many false conflicts in files with comparative analysis sections.

### Pitfall 4: Router Error Message Not Updated

**What goes wrong:** Adding new subcommands to the `research` namespace but forgetting to update the error message string in the `default` branch of the router's switch.
**Why it happens:** The error message at `src/router.js:1303` hardcodes available subcommands as a string.
**How to avoid:** Always update the error message in the router's default branch when adding new commands.
**Warning signs:** `bgsd-tools research:unknown` shows help text that doesn't list `score` or `gaps`.

### Pitfall 5: Build Not Run After Source Changes

**What goes wrong:** Source changes in `src/commands/research.js` aren't reflected when running `node bin/bgsd-tools.cjs` because the bundle is stale.
**Why it happens:** The built file (`bin/bgsd-tools.cjs`) is a bundled output from esbuild. Editing source requires `npm run build`.
**How to avoid:** Always run `npm run build` after editing source files, then test with `node bin/bgsd-tools.cjs`. Tests import from `src/` directly for unit tests, but CLI integration tests use the built bundle.
**Warning signs:** Test passes but CLI command doesn't work (or vice versa).

## Open Questions

1. **Conflict detection scope:** The CONTEXT.md says conflict detection sensitivity is at the agent's discretion. Current plan is to scan for explicit disagreement language near source citations. However, RESEARCH.md files to date contain very few actual multi-source conflicts (the template encourages prescriptive recommendations, not comparison). The `conflicts[]` array may often be empty â€” this is expected and correct.

2. **`research:score` for project-level research files:** The milestone workflow produces `STACK.md`, `FEATURES.md`, `ARCHITECTURE.md`, `PITFALLS.md` in `.planning/research/` â€” these follow a different template than phase-level `*-RESEARCH.md` files. The score command should handle both, or at minimum not crash on project-level research files. Defensive parsing (fallback to empty/null for missing sections) handles this naturally.

3. **`oldest_source_days` from research date vs today:** The profile field `oldest_source_days` computes the age of the research itself (days since `**Researched:** YYYY-MM-DD`), not the age of individual cited sources. This is a pragmatic choice â€” individual source ages would require date parsing from URLs and descriptions, which is unreliable.
