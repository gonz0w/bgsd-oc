# Requirements

**Milestone:** v17.1 Workflow Reliability & Foundation Hardening
**Generated:** 2026-03-30
**Status:** Drafted

## Goal

Improve workflow execution reliability by turning recurring lessons and the codebase efficiency audit into concrete fixes for JJ-aware execution, verifier fidelity, shared planning contracts, and plan-scoped metadata updates.

## Requirement Categories

### FOUND - Shared Contracts and Safe Mutation

- [x] `FOUND-01` User can rely on state and session mutations using one canonical update path for the touched milestone flows, so Markdown and SQLite views do not drift during normal operation.
- [x] `FOUND-02` User can run concurrent state, memory, or handoff-affecting commands without silent lost updates from lock-free whole-file rewrites.
- [x] `FOUND-03` User can run verification and planning analyzers that reuse shared planning or workspace indexes instead of rescanning the same project data per item.
- [x] `FOUND-04` Maintainer can update state/config/storage contracts in one shared implementation for touched CLI and plugin surfaces instead of patching duplicate logic.

### EXEC - JJ-Aware Execution Reliability

- [x] `EXEC-01` User can complete task and metadata commits safely in detached or dirty JJ colocated workspaces using a supported path-scoped flow.
- [x] `EXEC-02` User gets an automatic or clearly documented JJ-native fallback when generic commit helpers reject a valid path-scoped execution state.
- [x] `EXEC-03` Maintainer can verify runtime-sensitive source changes against the correct local or rebuilt runtime so stale bundled artifacts do not hide real behavior.

### VERIFY - Verifier Fidelity and Evidence Quality

- [x] `VERIFY-01` User can verify plans whose `must_haves` metadata uses nested, inline-array, or plain-string shapes without false "missing metadata" failures.
- [x] `VERIFY-02` User gets a loud, actionable verifier result when artifact or key-link extraction is empty or inconclusive instead of a misleading success response.
- [x] `VERIFY-03` User gets verification results based on repo-local current-state evidence when validating source changes in the development workspace.
- [x] `VERIFY-04` User gets broader guidance-surface verification when a phase success criterion depends on command-family or discoverability outcomes beyond the directly touched test file.

### PLAN - Planner and Checker Execution Realism

- [x] `PLAN-01` Planner and checker flows validate verifier-consumable plan metadata before approval so malformed `must_haves` blocks are caught before execution.
- [x] `PLAN-02` Planner and checker flows catch stale command examples, stale file paths, and unavailable validation steps before handing plans to execution.
- [x] `PLAN-03` Planner and checker flows catch task-order verification dependencies or overscoped batches early enough to avoid execution-time repair loops.

### STATE - Plan-Scoped Summary and Completion Accuracy

- [x] `STATE-01` User gets summary file lists and metrics derived from plan-scoped truth such as task commits or declared plan files rather than the ambient dirty working copy.
- [x] `STATE-02` User gets refreshed plan totals, current-plan fields, and focus text after automated completion updates so STATE and roadmap summaries reflect on-disk reality.
- [x] `STATE-03` User gets readback validation for automated metadata updates so stale high-level summary fields are patched before finalization.

### LOG - Logging Contract and Default-Noise Reduction

- [x] `LOG-01` User gets one consistent debug or verbosity contract across CLI and plugin diagnostics for touched flows.
- [x] `LOG-02` User gets clean default stdout and quieter normal-path stderr output, with routine debug chatter and duplicate diagnostics moved behind explicit verbosity controls.

## Future Requirements

- Explore dynamic model configuration after workflow hardening reduces current execution noise.
- Revisit CMUX-first UX after planning and execution state become more deterministic.
- Consider deeper module-boundary cleanup beyond milestone-critical surfaces once shared contracts are in place.

## Out of Scope

- Changes to how lessons are captured, stored, or managed as a product feature
- Broad architectural rewrites without milestone-scoped acceptance criteria
- New user-facing product features unrelated to workflow reliability hardening
- Speculative performance work without repeated-work or reliability evidence
