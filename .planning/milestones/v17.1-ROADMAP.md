# Roadmap: bGSD Plugin

## Milestones

- ✅ **v17.0 JJ Workspaces, Intent Cascade & Command Simplification** - Phases 155-162 (shipped 2026-03-30)
- 🚧 **v17.1 Workflow Reliability & Foundation Hardening** - Phases 163-167 (active)

## Overview

v17.1 hardens the workflow foundation that recent lessons and the reliability audit keep surfacing: shared state/config/storage contracts come first, then planning and verifier correctness, then JJ-aware execution and repo-local runtime truth, then plan-scoped completion accuracy, and finally one quieter, more predictable logging contract. The milestone stays inside the existing Node.js + SQLite + JJ + Markdown architecture and focuses on reliability, drift prevention, and repeated-work reduction rather than new product features or broad rewrites.

## Phases

- [x] **Phase 163: Shared Contracts & Safe Mutation** - Establish one canonical mutation contract for touched state/config/storage flows and make concurrent updates coordination-safe. (completed 2026-03-30)
- [x] **Phase 164: Shared Planning Indexes & Metadata Truthfulness** - Reuse shared planning indexes and make planner/verifier metadata handling match real `must_haves` shapes. (completed 2026-03-30)

## Phase Details

### Phase 163: Shared Contracts & Safe Mutation
**Goal**: Users can rely on one canonical update contract for touched milestone flows, and concurrent workflow mutations no longer silently drift or overwrite each other
**Depends on**: Phase 162
**Requirements**: FOUND-01, FOUND-02, FOUND-04
**Success Criteria** (what must be TRUE):
  1. Touched state and session mutations update Markdown and SQLite through the same canonical path instead of producing drift during normal workflow execution.
  2. Concurrent state, memory, or handoff-affecting commands no longer silently lose updates from lock-free whole-file rewrites.
  3. Maintainers can change shared state/config/storage behavior for touched flows in one implementation rather than patching duplicate CLI and plugin paths.
  4. Existing planning artifacts remain readable and writable through the hardened mutation path without compatibility regressions.
**Plans**: 5/5 plans complete

### Phase 164: Shared Planning Indexes & Metadata Truthfulness
**Goal**: Planning and verification flows consume real plan metadata through shared indexes and fail loudly when evidence is missing or inconclusive
**Depends on**: Phase 163
**Requirements**: FOUND-03, VERIFY-01, VERIFY-02, PLAN-01
**Success Criteria** (what must be TRUE):
  1. Verifier accepts nested, inline-array, and plain-string `must_haves` metadata shapes used by real plans instead of falsely reporting missing metadata.
  2. Verifier returns a loud, actionable failure when artifact or key-link extraction is empty or inconclusive rather than reporting misleading success.
  3. Planner and checker flows reject malformed verifier-facing `must_haves` metadata before plans are approved for execution.
  4. Repeated planning and verification analysis reuses shared planning or workspace indexes instead of rescanning the same project data per item.
**Plans**: 3/3 plans complete

### Phase 165: JJ Execution & Repo-Local Verification
**Goal**: Users can execute and verify workflow changes reliably from real JJ workspace states without stale runtime confusion
**Depends on**: Phase 164
**Requirements**: EXEC-01, EXEC-02, EXEC-03, VERIFY-03
**Success Criteria** (what must be TRUE):
  1. Users can complete task and metadata commits safely from supported dirty or detached JJ colocated workspace states using a path-scoped flow.
  2. When generic commit helpers reject a valid JJ execution state, users get an automatic or clearly documented JJ-native fallback that still completes the intended commit.
  3. Runtime-sensitive source changes are checked against the correct local or rebuilt runtime so stale bundled artifacts do not hide real behavior.
  4. Verification results in the development workspace are based on repo-local current-state evidence that matches the active checkout.
**Plans**: 3/3 plans complete

### Phase 166: Execution Realism & Plan-Scoped Completion
**Goal**: Planned work stays executable as written, and automated completion outputs reflect the active plan instead of ambient workspace noise
**Depends on**: Phase 165
**Requirements**: VERIFY-04, PLAN-02, PLAN-03, STATE-01, STATE-02, STATE-03
**Success Criteria** (what must be TRUE):
  1. Planner and checker flows catch stale command examples, stale file paths, unavailable validation steps, and task-order verification dependencies before execution begins.
  2. Overscoped task batches are flagged early enough to avoid routine execution-time repair loops.
  3. Verification expands to guidance and discoverability surfaces when a phase success criterion depends on command-family outcomes beyond the directly touched test file.
  4. Summary file lists and completion metrics come from plan-owned commits or declared plan files rather than unrelated dirty working-copy changes.
  5. Automated completion updates read back and repair plan totals, current-plan fields, focus text, and related summary metadata so STATE and roadmap summaries match on-disk reality.
**Plans**: 2/2 plans complete

### Phase 167: Logging Contract & Quiet Defaults
**Goal**: Users get one predictable diagnostic contract with quiet defaults and explicit verbosity for reliability investigations
**Depends on**: Phase 166
**Requirements**: LOG-01, LOG-02
**Success Criteria** (what must be TRUE):
  1. Touched CLI and plugin reliability flows use one shared debug or verbosity contract instead of conflicting diagnostics rules.
  2. Successful default runs keep stdout clean and routine debug chatter out of normal stderr output.
  3. Enabling verbosity exposes the diagnostics needed to investigate failures without duplicate or contradictory messages.
**Plans**: 2/2 plans complete
