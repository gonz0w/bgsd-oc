  <name>Task 1: Create decision-rules.js with pure decision functions and registry</name>
  <files>src/lib/decision-rules.js</files>
  <action>
Create `src/lib/decision-rules.js` as a CJS module (following `src/lib/` convention: `module.exports`).

Implement ~12 pure decision functions based on the RESEARCH.md analysis of 85 offloadable candidates grouped into unique decision types. Every function follows the same contract: receives a state object, returns `{value, confidence, rule_id, metadata?}`.

**Decision functions to implement (from RESEARCH.md §Decision Candidate Analysis):**

1. `resolveContextGate(state)` — bgsd-context presence check. Input: `{context_present}`. Returns `{value: boolean, confidence: 'HIGH', rule_id: 'context-gate'}`. Covers ~30 workflow locations.

2. `resolveProgressRoute(state)` — Progress workflow Routes A-F. Input: `{plan_count, summary_count, uat_gap_count, current_phase, highest_phase, roadmap_exists, project_exists, state_exists}`. Returns route letter with HIGH confidence (MEDIUM for between-milestones fallback). Covers 5 locations.

3. `resolveResumeRoute(state)` — Resume project next-action. Input: `{has_state, has_roadmap, has_plans, has_incomplete_plans, has_blockers, phase_complete}`. Returns action string. Use MEDIUM confidence (state can be ambiguous). Covers 5 locations.

4. `resolveExecutionPattern(state)` — Execute-plan Pattern A/B/C. Input: `{task_types}`. Returns pattern letter with HIGH confidence. Covers 1 location.

5. `resolveContextBudgetGate(state)` — Context budget warning check. Input: `{warning, mode}`. Returns `{value: 'proceed'|'warn'|'stop', confidence: 'HIGH'}`. Covers 1 location.

6. `resolvePreviousCheckGate(state)` — Previous SUMMARY blocker check. Input: `{has_previous_summary, has_unresolved_issues, has_blockers}`. Returns `{value: 'proceed'|'warn'|'block', confidence: 'HIGH'}`. Covers 1 location.

7. `resolveCiGate(state)` — CI quality gate check. Input: `{ci_enabled, has_test_command, tests_passing}`. Returns `{value: 'run'|'skip'|'warn', confidence: 'HIGH'}`. Covers 2 locations.

8. `resolvePlanExistenceRoute(state)` — Phase has plans / needs planning. Input: `{plan_count, has_research, has_context}`. Returns routing advice. HIGH confidence. Covers 4 locations.

9. `resolveBranchHandling(state)` — Branch merge strategy. Input: `{branching_strategy, has_branch, branch_behind}`. Returns strategy. MEDIUM confidence (user preference dependent). Covers 1 location.

10. `resolveAutoAdvance(state)` — Auto-advance config check. Input: `{auto_advance_config, auto_flag}`. Returns `{value: boolean, confidence: 'HIGH'}`. Covers 2 locations.

11. `resolvePhaseArgParse(state)` — Phase number from arguments. Input: `{raw_arg}`. Returns `{value: number|null, confidence: 'HIGH'}`. Covers 3 locations.

12. `resolveDebugHandlerRoute(state)` — Debug return handling. Input: `{return_type}`. Returns `{value: 'fix'|'plan'|'manual'|'continue', confidence: 'MEDIUM'}`. Covers 2 locations.

**Registry:** Create `DECISION_REGISTRY` array where each entry has: `{id, name, category, description, inputs, outputs, confidence_range, resolve}`. Categories: workflow-routing, execution-mode, state-assessment, template-selection.

**Aggregator:** Create `evaluateDecisions(command, state)` function that evaluates all applicable rules for a given command context and returns a map of `{rule_id: result}`. This is the function the enricher will call in Plan 02.

Follow the pattern from `resolveModelInternal()` in helpers.js — check overrides first, then lookup tables, then fallback. Use no file I/O in any decision function (they receive pre-computed state).

**Test structure:**

1. **Registry completeness tests:**
   - Registry has >= 10 entries
   - Every entry has required fields: id, name, category, description, inputs, outputs, confidence_range, resolve
   - Every entry's resolve is a function
   - No duplicate rule IDs

2. **Contract tests for each decision function:**
   - Call with valid state → returns `{value, confidence, rule_id}`
   - `confidence` is one of HIGH, MEDIUM, LOW
   - `rule_id` matches the registry entry's id
   - `value` is within expected range (e.g., progress route returns A-F or no-project/no-state)

3. **Edge case tests:**
   - resolveProgressRoute with every major state combination (each route A-F)
   - resolveExecutionPattern with checkpoint types
   - resolveResumeRoute with ambiguous state (returns MEDIUM)
   - Functions handle missing/undefined state fields gracefully (no throw)

4. **evaluateDecisions aggregator test:**
   - Returns object with rule_id keys and decision result values
   - Only includes rules applicable to given command context

5. **Confidence distribution test:**
   - Verify not all rules return HIGH — at least 2 entries in registry have 'MEDIUM' in confidence_range
