<feature>
  <name>Resumable release mutations with safe cleanup boundaries and github-ci compatibility</name>
  <files>src/lib/release/mutate.js, src/lib/release/state.js, src/lib/release/pr.js, tests/release.test.cjs</files>
  <behavior>
Applied releases should update version files, create tags/PRs, remain compatible with the existing github-ci path, and recover from partial failure without magical rollback.

Cases:
  - applying a confirmed release → package.json, package-lock.json, and VERSION stay in sync
  - tag creation succeeds but later step fails → `.planning/release-state.json` records the next safe resume step
  - gh unavailable or unauthenticated → PR step stops with explicit guidance before unsafe follow-on work
  - release PR created successfully → resulting branch and PR metadata stay usable by the existing github-ci workflow instead of forking a separate CI path
  - safe cleanup is obvious (for example removing an unpushed local tag) → cleanup may run automatically
  - meaningful content edits already happened → preserve files and provide resume guidance instead of rewriting them
  </behavior>
  <implementation>
Persist release progress inside `.planning/release-state.json`, separate mutation helpers from analysis helpers, integrate git and gh preflight checks early, keep created PR metadata compatible with the repo's existing github-ci flow, and encode cleanup boundaries so only obviously safe rollback actions happen automatically.
  </implementation>
</feature>
