# Roadmap: bGSD Plugin

## Milestones

- ✅ **v16.1 Workflow Reliability & Acceleration** - Phases 149-154 (shipped 2026-03-29)
- 🚧 **v17.0 JJ Workspaces, Intent Cascade & Command Simplification** - Phases 155-162 (active)

## Overview

v17.0 makes JJ-first execution the required local execution model, replaces Git worktree assumptions with JJ workspace-backed parallelism and recovery, reduces the visible slash-command surface to a smaller canonical set, and cascades intent through milestone and phase planning with compact `effective_intent` injection. The milestone starts by enforcing JJ-backed execution and shipping the JJ-native workspace lifecycle, then hardens parallel wave recovery and planning context, then flips command discoverability to grouped canonical families with audited references, and finishes by making phase intent explicit and verification intent-aware. Dynamic model configuration remains explicitly deferred beyond this milestone.

## Phase Details

### Phase 155: JJ Execution Gate & Workspace Lifecycle
**Goal**: Users can only start bGSD execution from a JJ-backed repo and can manage local execution isolation through JJ-native workspace commands instead of Git worktrees
**Depends on**: Phase 154
**Requirements**: JJ-01, JJ-02
**Success Criteria** (what must be TRUE):
  1. Starting execution or execution-oriented init flows in a Git-only repo fails with a clear JJ-required message and points users to the supported setup path
  2. Users can create, inspect, and clean up execution workspaces through JJ-native commands rather than worktree-shaped commands
  3. The supported local parallel execution path is visibly JJ-first across execution entrypoints, not a Git worktree fallback
**Plans**: 2/2 plans complete

### Phase 156: JJ Parallel Waves, Recovery & Coverage
**Goal**: Users can run parallel plan waves in JJ workspaces, recover from stale or divergent runs, and rely on JJ-first execution guidance throughout the repo
**Depends on**: Phase 155
**Requirements**: JJ-03, JJ-04, JJ-06
**Success Criteria** (what must be TRUE):
  1. Multi-plan execution creates, tracks, and reconciles one JJ workspace per runnable plan in a wave
  2. When a workspace becomes stale or divergent, users get JJ operation-log-backed diagnostics and a supported recovery path instead of merge-back guidance
  3. Users can inspect, recover, and clean up failed JJ workspace runs without manually reconstructing what happened
  4. Maintainers can read config/help/docs and run tests that validate JJ workspace lifecycle, stale handling, and recovery as the supported execution model
**Plans**: 3/3 plans complete

### Phase 157: Planning Context Cascade
**Goal**: Milestone planning flows receive compact layered intent plus JJ workspace capability context so roadmap and plan design stay aligned with both purpose and safe parallelism
**Depends on**: Phase 156
**Requirements**: JJ-05, INT-01, INT-03, INT-04, INT-05
**Success Criteria** (what must be TRUE):
  1. The active milestone has a lightweight `MILESTONE-INTENT.md` capturing why it matters now, targeted outcomes, priorities, and non-goals
  2. New milestone and roadmap/planning flows keep milestone strategy in milestone intent instead of mutating project intent for temporary priorities
  3. Roadmapper, planner, researcher, verifier, and verify-work flows receive a compact `effective_intent` object rather than raw pasted intent documents
  4. Planning and roadmapping surfaces can see JJ-required, workspace-backed execution capability context and use it to prefer safe sibling work when overlap is low
**Plans**: 5/5 plans complete

### Phase 158: Canonical Command Families
**Goal**: Users can drive bGSD through a smaller canonical command set while existing entrypoints continue to work during migration
**Depends on**: Phase 157
**Requirements**: CMD-01, CMD-02, CMD-03
**Success Criteria** (what must be TRUE):
  1. `/bgsd-quick` is the clearly recommended quick-entry path while `/bgsd-quick-task` still resolves as a compatibility alias
  2. Planning, roadmap, todo, settings, and advanced inspection flows are available through grouped canonical command families instead of multiple sibling top-level commands
  3. Legacy slash commands still route successfully during migration, but surfaced guidance consistently prefers the canonical grouped forms
  4. Users reaching the same workflow through canonical commands or legacy aliases land in equivalent underlying behavior
**Plans**: 15/15 plans executed; verification gaps remain

### Phase 159: Help Surface & Command Integrity
**Goal**: Users can trust bGSD help and next-step guidance to be smaller, current, and executable exactly as written
**Depends on**: Phase 158
**Requirements**: CMD-04, CMD-05, CMD-06
**Success Criteria** (what must be TRUE):
  1. The main help surface clearly separates core vs advanced commands and includes `/bgsd-review` in the primary command path
  2. Automated command-reference validation catches nonexistent commands, stale preferred aliases, and incorrect next-step guidance in user-facing surfaces
  3. Workflow, template, skill, agent, and doc examples include the arguments and flags required for the intended flow
  4. Users following surfaced guidance for planning, gap closure, settings, and execution do not have to guess missing parameters to reach the right path
**Plans**: 20/20 plans complete

### Phase 160: Phase Intent & Alignment Verification
**Goal**: Each phase carries explicit local intent and verification can judge both what shipped and whether it matched the phase purpose
**Depends on**: Phase 157, Phase 159
**Requirements**: INT-02, INT-06
**Success Criteria** (what must be TRUE):
  1. Phase context artifacts include a lightweight intent section describing local purpose, expected user change, and non-goals without creating a separate top-level phase intent file
  2. Verification and UAT outputs report requirement coverage and intent alignment as separate judgments for the current phase
  3. Planners and verifiers can explain what user-facing change a phase is meant to unlock, not only which requirements it traces to
**Plans**: 2/2 plans complete

### Phase 161: Canonical Handoff Guidance & Phase 158 Closure
**Goal**: Generated planning-prep handoffs, restart guidance, and parity coverage all use canonical planning-family commands so the remaining Phase 158 command-migration blocker can close cleanly
**Depends on**: Phase 159, Phase 160
**Requirements**: CMD-02, CMD-03, CMD-05, CMD-06
**Success Criteria** (what must be TRUE):
  1. Generated discuss/research/repair handoffs emit canonical `/bgsd-plan discuss|research|assumptions` routes instead of legacy planning-prep aliases
  2. Resume and repair output stays executable as written with the required phase arguments and remains covered by command-integrity validation
  3. Phase 158 re-verification can confirm the remaining planning-prep alias blocker is gone without regressing canonical-vs-alias parity behavior
**Plans**: 1/1 plans complete

### Phase 162: Intent Runtime Parity & Legacy Context Stability
**Goal**: Legacy phase contexts without the explicit intent block resolve the same way in source and shipped runtime so verification and UAT intent-alignment results stay stable across rebuilds
**Depends on**: Phase 160
**Requirements**: INT-02, INT-06
**Success Criteria** (what must be TRUE):
  1. Source and bundled runtime share one explicit fallback for legacy phase contexts missing the new local-intent block
  2. Verification and UAT intent-alignment outputs stay consistent after rebuilds and continue to report `not assessed` when legacy context lacks explicit intent
  3. Regression proof covers the runtime/source parity path so future intent-parser edits cannot silently change shipped alignment behavior
**Plans**: 2/2 plans complete
