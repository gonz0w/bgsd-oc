## Category: JJ (JJ-First Execution)

### Requirements

- [x] **JJ-01:** JJ required for execution - Execution workflows and init surfaces fail clearly when the repo is not initialized for JJ-backed execution

- [x] **JJ-02:** Workspace lifecycle replaces worktrees - bGSD exposes JJ-native workspace lifecycle commands and removes Git worktree execution commands as the supported parallel backend

- [x] **JJ-03:** Parallel execution uses JJ workspaces - Multi-plan execution waves create, track, and reconcile one JJ workspace per runnable plan instead of Git worktrees

- [x] **JJ-04:** Operation-log recovery and stale handling - Execution flows detect stale workspaces and provide JJ operation-log-backed diagnostics and recovery helpers for failed or divergent runs

- [x] **JJ-05:** Planner and roadmapper are workspace-aware - Planning and roadmapping flows receive compact JJ capability context so they can prefer safe parallel sibling work when overlap is low

- [x] **JJ-06:** JJ-first docs, config, and tests - Config, docs, and test coverage describe JJ-first execution and validate workspace lifecycle, stale-workspace handling, and recovery behavior

---

## Category: CMD (Command Footprint Reduction)

### Requirements

- [x] **CMD-01:** Canonical quick entry - `/bgsd-quick` is the single recommended quick-entry command and `/bgsd-quick-task` is treated as a compatibility alias only

- [x] **CMD-02:** Canonical grouped command families - Planning, roadmap, todo, settings, and advanced inspection flows use grouped canonical command families instead of multiple sibling top-level commands

- [x] **CMD-03:** Backward-compatible alias routing - Legacy slash commands continue to resolve during migration without remaining the preferred help or docs path

- [x] **CMD-04:** Core help surface refresh - Main help output teaches the reduced canonical command set, clearly separates core vs advanced paths, and includes `/bgsd-review`

- [x] **CMD-05:** Command reference integrity audit - The repo has an auditable scan or validation path that catches nonexistent commands, stale preferred aliases, and incorrect next-step guidance in user-facing surfaces

- [x] **CMD-06:** Parameter-correct next steps - Workflows, templates, skills, agents, and docs suggest command examples with the required arguments and flags for the intended flow

---

## Category: INT (Intent Cascade)

### Requirements

- [x] **INT-01:** Milestone intent layer - The active milestone has a lightweight `.planning/MILESTONE-INTENT.md` that captures why the milestone matters now, targeted outcomes, priorities, and non-goals

- [x] **INT-02:** Phase intent in context - Phase context artifacts include a lightweight intent section describing local purpose, expected user change, and non-goals without adding a separate top-level phase intent file

- [x] **INT-03:** Effective intent computation - bGSD computes a compact `effective_intent` object by layering project, milestone, and phase intent instead of injecting raw intent documents

- [x] **INT-04:** Planning and verification injection - Roadmapper, planner, researcher, verifier, and verify-work flows receive the computed `effective_intent` where purpose and alignment matter

- [x] **INT-05:** Milestone workflow stops overloading project intent - New milestone flows review project intent but track milestone-specific strategy in milestone intent rather than mutating project intent to hold temporary priorities

- [x] **INT-06:** Intent alignment verification - Verification and UAT flows can report requirement coverage separately from intent alignment for the current phase
