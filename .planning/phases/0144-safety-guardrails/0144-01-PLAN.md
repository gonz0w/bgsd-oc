  <name>Task 1: Add GARD-04 pattern library, Unicode normalization, and container detection to advisory-guardrails.js</name>
  <files>src/plugin/advisory-guardrails.js</files>
  <action>
Extend `src/plugin/advisory-guardrails.js` with the GARD-04 destructive command detection system. All additions follow the existing factory-function pattern established by GARD-01/02/03.

**Module-level additions (after WRITE_TOOLS):**

1. Add `BASH_TOOLS` set: `new Set(['bash'])` — parallel to existing `WRITE_TOOLS`.

2. Add `DESTRUCTIVE_PATTERNS` array with 25+ patterns organized by category. Each entry: `{ id, category, severity, pattern, description }`. Structure mirrors `SECURITY_PATTERNS` from `src/commands/skills.js`. Categories and patterns per CONTEXT.md:
   - **filesystem** (5): `fs-rm-recursive` (critical: `rm -r`/`rm -rf`/`rm --recursive`), `fs-rm-force` (warning: `rm -f` without recursive), `fs-rm-plain` (info: plain `rm`), `fs-format` (critical: `mkfs`/`format`), `fs-dd` (critical: `dd of=`)
   - **database** (3): `db-drop-table` (critical: DROP TABLE/DATABASE/SCHEMA), `db-truncate` (critical: TRUNCATE TABLE), `db-delete-no-where` (warning: DELETE FROM without WHERE)
   - **git** (4): `git-force-push` (critical: `--force`), `git-force-push-f` (critical: `-f` shorthand), `git-reset-hard` (warning), `git-clean-fd` (warning)
   - **system** (7): `sys-kill-9` (warning), `sys-chmod-777` (warning), `sys-chmod-recursive` (warning), `sys-chown-recursive` (warning), `sys-shutdown` (critical: shutdown/reboot/halt/poweroff), `sys-iptables-flush` (critical), `sys-systemctl-disable` (warning: `systemctl disable`/`systemctl stop`)
   - **supply-chain** (6): `sc-curl-pipe` (info: `curl | sh`), `sc-wget-pipe` (info: `wget | sh`), `sc-eval` (info: `eval`), `sc-npm-global` (info: `npm install -g`), `sc-pip-sudo` (info: `sudo pip install`), `sc-source-remote` (info: `source <(curl ...)` or `. <(curl ...)`)

   Use `\b` word boundaries to prevent false positives. Use the exact regex patterns from RESEARCH.md Example 1 as the starting point, extending to reach 25+ total.

3. Add `normalizeCommand(raw)` function — three-step Unicode normalization pipeline per CONTEXT.md:
   - Step 1: `raw.normalize('NFKD')` — NFKD decomposition (handles fullwidth chars, ligatures)
   - Step 2: Strip zero-width characters: `replace(/[\u200B-\u200D\u2060\uFEFF]/g, '')`
   - Step 3: Strip combining marks: `replace(/[\u0300-\u036F]/g, '')`

4. Add `detectSandboxEnvironment(configOverride)` function:
   - If `configOverride === true` return true; if `configOverride === false` return false
   - Check env vars: `DOCKER_HOST`, `SINGULARITY_NAME`, `MODAL_TASK_ID`, `DAYTONA_WS_ID`, `CODESPACES`, `GITPOD_WORKSPACE_ID`
   - Filesystem probes (wrapped in try/catch): `/.dockerenv`, `/run/.containerenv`, `/proc/self/cgroup` containing docker|containerd|kubepods
   - Return boolean
   - Import `statSync` and `readFileSync` from 'fs' (readFileSync already imported at top)

5. Add `mergePatterns(builtIn, custom)` function:
   - Iterate custom patterns from config, validate required fields (id, pattern, category, severity)
   - Convert string patterns to RegExp via `new RegExp(pattern)` wrapped in try/catch (skip invalid regex with logger warning)
   - Return merged array `[...builtIn, ...validCustom]`
   - Custom patterns get `custom: true` flag

6. Add `matchPatterns(normalizedCommand, patterns, disabledPatterns, categoryConfig)` function:
   - Filter out patterns whose `id` is in `disabledPatterns` set
   - Filter out patterns whose `category` is disabled in `categoryConfig`
   - Test remaining patterns against normalizedCommand
   - Return array of matching pattern objects (could be multiple matches per command)

**Factory function additions (inside `createAdvisoryGuardrails`):**

7. Read `destructive_commands` config from `guardConfig.destructive_commands || {}`:
   - `destructiveEnabled = destructiveConfig.enabled !== false`
   - `sandboxMode = destructiveConfig.sandbox_mode ?? 'auto'`
   - `categoryConfig = destructiveConfig.categories || {}`
   - `disabledPatterns = new Set(destructiveConfig.disabled_patterns || [])`
   - `customPatterns = destructiveConfig.custom_patterns || []`

8. At init time: detect sandbox via `detectSandboxEnvironment(sandboxMode)`, cache as `isSandbox`.

9. At init time: merge patterns via `mergePatterns(DESTRUCTIVE_PATTERNS, customPatterns)`, cache as `mergedPatterns`.

**onToolAfter additions (BEFORE the existing WRITE_TOOLS check):**

10. Add GARD-04 block:
    - Check `destructiveEnabled && BASH_TOOLS.has(toolName)`
    - Extract `input?.args?.command` (NOT `input?.args?.filePath` — bash tool uses `args.command`)
    - If no command, return
    - Normalize: `normalizeCommand(rawCommand)`
    - Match: `matchPatterns(normalized, mergedPatterns, disabledPatterns, categoryConfig)`
    - For each match:
      - In sandbox mode (`isSandbox`): skip if `match.severity !== 'critical'`
      - Build behavioral guidance string per severity (CRITICAL: "Confirm with user before proceeding with this destructive operation.", WARNING: "Proceed with caution — this operation may be difficult to reverse.", INFO: empty)
      - Call `notifier.notify()` with:
        - `type: 'advisory-destructive'`
        - `severity: 'info'` — ALWAYS 'info' for notification routing to ensure context-only delivery (no OS notifications). Per RESEARCH.md Open Question 1, option (a): use severity 'info' for all GARD-04 notifications, embed logical severity in message text.
        - `message: \`GARD-04: ${rawCommand.slice(0, 80)} matched [${match.id}] (${match.severity.toUpperCase()})${behavioral ? '. ' + behavioral : ''}\``
    - `return;` after GARD-04 block to prevent fall-through to GARD-01/02/03

**Key decision per CONTEXT.md stress test:** Notification routing uses `severity: 'info'` for ALL GARD-04 notifications so they route to context injection only (never OS notifications). The logical severity (CRITICAL/WARNING/INFO) is embedded in the message text for LLM behavioral guidance.
