# Plan 01: Scaffold Merge Library & Data/Judgment Infrastructure

## Objective

Create the reusable scaffold merge library that both `plan:generate` and `verify:generate` will depend on. This extracts and generalizes the data/judgment separation pattern from `summary:generate` (JUDGMENT_SECTIONS + `mergeSummary()` in `src/commands/misc.js`) into a standalone `src/lib/scaffold.js` module with well-defined constants and merge functions.

## Tasks

### Task 1: Create scaffold merge library

- **Type:** auto
- **Files:**
  - `src/lib/scaffold.js` (create)

- **Action:**

  1. **Define marker constants:**
     - `DATA_MARKER = '<!-- data -->'` — marks sections pre-filled by CLI from deterministic sources
     - `JUDGMENT_MARKER = '<!-- judgment -->'` — marks sections the LLM fills with reasoning/assessment
     - `DATA_END = '<!-- /data -->'` and `JUDGMENT_END = '<!-- /judgment -->'` — closing markers

  2. **Implement `markSection(content, type)`** — wraps content block with the appropriate marker pair. `type` is `'data'` or `'judgment'`. Returns `<!-- data -->\ncontent\n<!-- /data -->` (or judgment equivalent).

  3. **Implement `parseMarkedSections(text)`** — parses a scaffold document (with frontmatter already stripped) into a structured map:
     - Splits on `## Heading` boundaries (same regex as `parseSummarySections` in misc.js line 2470)
     - For each section, detects whether it contains `<!-- data -->` or `<!-- judgment -->` marker
     - Returns `Map<heading, { content: string, markerType: 'data'|'judgment'|null }>` 
     - Handles sections with no markers (legacy/manual content) gracefully

  4. **Implement `mergeScaffold(existingText, freshText, judgmentHeadings)`** — idempotent merge following the `mergeSummary()` pattern (misc.js lines 2364-2464):
     - Parse both documents into section maps via `parseMarkedSections()`
     - For each heading in `judgmentHeadings`: if the existing section does NOT contain `TODO:`, preserve it (LLM already filled it)
     - For all other sections (data sections): always use the fresh version
     - Preserve frontmatter fields that contain non-TODO values from existing (same list as summary merge: `key-decisions`, `one-liner`, etc.)
     - Rebuild the full document: frontmatter + data sections (fresh) + judgment sections (preserved or fresh)
     - Return the merged document string

  5. **Implement `formatFrontmatter(obj)`** — serialize a JS object to YAML frontmatter string. Reuse the `reconstructFrontmatter()` pattern from misc.js. Handle arrays, nested objects, and string quoting correctly.

  6. **Export all functions** via `module.exports` (CJS format matching project convention).

- **Verify:** `require('./src/lib/scaffold')` loads without error. Exported functions are `markSection`, `parseMarkedSections`, `mergeScaffold`, `formatFrontmatter`, `DATA_MARKER`, `JUDGMENT_MARKER`.

- **Done:** `src/lib/scaffold.js` exists with all 4 functions + 2 constants exported. The merge logic follows the proven `mergeSummary()` pattern but is generalized for any document type with configurable judgment headings.

### Task 2: Unit tests for scaffold merge library

- **Type:** auto
- **Files:**
  - `tests/scaffold.test.cjs` (create)

- **Action:**

  1. **Test `markSection()`:**
     - `markSection('content', 'data')` produces `<!-- data -->\ncontent\n<!-- /data -->`
     - `markSection('content', 'judgment')` produces `<!-- judgment -->\ncontent\n<!-- /judgment -->`

  2. **Test `parseMarkedSections()`:**
     - Parse a document with mixed data/judgment sections → correct map entries
     - Parse a document with no markers → all sections have `markerType: null`
     - Parse empty document → empty map

  3. **Test `mergeScaffold()` — fresh generation:**
     - When `existingText` is null/undefined, return `freshText` unchanged
     - When existing has only TODO content in judgment sections, replace with fresh

  4. **Test `mergeScaffold()` — preserve judgment:**
     - Existing has LLM-written content (no TODO) in a judgment section → preserved
     - Existing has TODO in judgment section → replaced with fresh
     - Data sections always replaced regardless of content

  5. **Test `mergeScaffold()` — idempotency:**
     - `mergeScaffold(result, fresh, headings)` where `result = mergeScaffold(existing, fresh, headings)` produces identical output

  6. **Test `formatFrontmatter()`:**
     - Objects with string values, arrays, nested objects serialize correctly
     - Output matches YAML format expected by the project's frontmatter parser

  7. Use `node:test` and `node:assert` (project convention per `tests/summary-generate.test.cjs`).

- **Verify:** `node --test tests/scaffold.test.cjs` passes all tests. Zero failures.

- **Done:** Comprehensive test suite covering marker operations, section parsing, merge behavior (fresh, preserve, idempotent), and frontmatter formatting.
