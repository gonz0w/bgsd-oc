## Tasks

<!-- judgment -->
### Task 1: Dangling reference check after elision

- **Type:** auto
- **Files:**
  - `src/plugin/command-enricher.js` (modify)
  - `tests/enricher-elision.test.cjs` (modify)

- **Action:**

  Extend `elideConditionalSections()` to perform a post-elision dangling reference scan:

  1. After stripping false conditional sections, collect the names of all elided sections into a list.
  2. Scan the remaining text for occurrences of each elided section name as a word (e.g., `tdd_execution`, `ci_quality_gate`). Use word-boundary matching to avoid false positives on substrings.
  3. If any dangling references are found, add them to the return value as `{ warnings: [{ section: 'name', references: ['line content'] }] }`.
  4. In the enricher integration (Task 3 from Plan 01), log dangling reference warnings to `console.error` when `BGSD_DEBUG` is set.
  5. Add dangling reference info to the `_elision` debug field in bgsd-context.

  This is a safety net — if someone adds a reference to `tdd_execution` in a non-conditional section, the check catches it when TDD is disabled.

  Add tests:
  - Text with dangling reference to elided section → warning reported
  - Text with no dangling references → clean result
  - Elided section name appears in a different conditional section (also elided) → no false positive

- **Verify:** `npm test` passes. Debug logging with `BGSD_DEBUG=1` shows dangling reference warnings when they exist.

- **Done:** Dangling reference scanner catches references to elided sections and reports them in debug output and `_elision` stats.

### Task 2: Cumulative token savings measurement

- **Type:** auto
- **Files:**
  - `src/commands/features.js` (modify — extend measureAllWorkflows or add new function)

- **Action:**

  Create a measurement that produces a cumulative savings table showing the reduction journey across milestones. This should be runnable as part of the phase verification process.

  1. **Take a current baseline**: Run `workflow:baseline` to capture post-elision token counts for all workflows.
  2. **Load Phase 134 baseline** (the original pre-compression baseline) from `.planning/baselines/` — find the earliest snapshot file.
  3. **Load Phase 135 baseline** (post-compression) from `.planning/baselines/` — find the snapshot taken after Phase 135 completed.
  4. **Generate savings table** for the three target workflows (execute-plan, execute-phase, quick) plus the 10 top workflows total:

  ```
  | Workflow | Original | Post-Compression | Post-Elision | Total Reduction |
  |----------|----------|------------------|--------------|-----------------|
  | execute-plan | 4,621 | 2,727 | 2,450 | -47.0% |
  | execute-phase | 5,632 | 3,321 | 3,100 | -44.9% |
  | quick | 2,812 | 1,659 | 1,500 | -46.7% |
  ```

  The post-elision column should show token counts with ALL conditional sections removed (worst-case savings, when all conditions are false — e.g., no TDD, no CI, no verification). This represents the maximum possible savings from elision.

  If Phase 134/135 baselines aren't available on disk, use the known token counts from Phase 135 VERIFICATION.md (hardcoded fallback):
  - execute-plan: pre=4621, post-compression=2727
  - execute-phase: pre=5632, post-compression=3321
  - quick: pre=2812, post-compression=1659

  Add a `workflow:savings` subcommand (or extend `workflow:compare`) that outputs this table as JSON and formatted text.

- **Verify:** Run the measurement command. Verify the table shows non-zero reduction in the post-elision column for workflows with conditional sections.

- **Done:** Cumulative savings table shows per-workflow token journey from original → compressed → elided with total reduction percentages. SC3 measurement requirement met.
