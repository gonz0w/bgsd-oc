<task type="auto">
  <name>Task 1: Integrate fd into discovery adapter as third file discovery mode</name>
  <files>src/lib/adapters/discovery.js</files>
  <action>
1. Add `const { findFiles: fdFindFiles, isFdAvailable } = require('../cli-tools');` and `const { isToolEnabled } = require('../cli-tools/fallback');` at the top of discovery.js.

2. Create a new `fdWalkSourceFiles(cwd, sourceDirs, skipDirs)` function:
   - For each source directory in `sourceDirs`, call `fdFindFiles('', { type: 'f', exclude: [...skipDirs].map(d => d), cwd })` using the fd wrapper.
   - fd respects .gitignore by default — no need for the ignore matcher.
   - Filter out BINARY_EXTENSIONS from results (same as optimizedWalkSourceFiles does).
   - Normalize paths using `normalizeRelativePath` and `fromPosix` (same post-processing as optimized mode).
   - Return the file list in the same format as `optimizedWalkSourceFiles`.
   
   IMPORTANT: The fd wrapper already handles fallback when fd unavailable — but for discovery.js, we want explicit control. Check `isToolEnabled('fd')` first. If fd is enabled and available, use fdWalkSourceFiles. Otherwise, fall through to the existing optimized (fast-glob) path. This way:
   - fd available + enabled → fd mode (fastest)
   - fd unavailable or disabled → optimized mode (fast-glob, existing)
   - optimized mode fails → legacy mode (readdirSync, existing)

3. Update `walkSourceFiles()` to try fd mode first:
   - Before the existing `runWithShadowCompare` call, check `isToolEnabled('fd')`.
   - If fd is enabled, run `fdWalkSourceFiles()` directly (no shadow compare for fd mode — it's an acceleration, not a replacement under parity testing).
   - If fd mode throws or returns empty results unexpectedly, fall through to existing optimized+legacy path.
   - The existing `runWithShadowCompare` remains untouched as the fallback.

4. Similarly create `fdGetSourceDirs(cwd)` to use fd for top-level source directory detection:
   - Use `fdFindFiles('', { type: 'd', maxDepth: 1, cwd })` to get top-level directories.
   - Filter through KNOWN_SOURCE_DIRS and SKIP_DIRS logic (same as optimizedGetSourceDirs).
   - For unknown dirs, use fd to check for source files inside (replace the fast-glob probe).

5. Update `getSourceDirs()` to try fd mode first, same pattern as walkSourceFiles.

Per CONTEXT.md: fd used ONLY for large-scale codebase file discovery. The .planning/ directory scanning and other small-scope operations continue using readdir/glob as before.

CRITICAL: The fd binary is `fdfind` on Debian/Ubuntu. The fd.js wrapper uses `execFileSync('fd', ...)` — Phase 124's detector resolves the correct binary name via `resolveToolPath()`, but the wrapper hardcodes `'fd'`. Check if the wrapper needs to use the resolved path. If the detector stores the resolved command in its result, use `detectTool('fd').command` or similar to get the correct binary name. If not available, the wrapper's `execFileSync('fd', ...)` will fail on Ubuntu but the fallback will catch it cleanly — acceptable for now since the Node.js fallback produces identical results.
  </action>

**Enhance jq.js fallback (src/lib/cli-tools/jq.js):**
1. Expand the JavaScript fallback in `transformJson()` to handle these additional filter patterns that are used in the codebase:
   - `select(condition)` — filter array elements matching a condition (e.g., `[.[] | select(.available == true)]`)
   - `map(expr)` — transform each element (e.g., `[.[] | .name]` or `map(.name)`)
   - Pipe chains — sequential filters separated by `|` (e.g., `.[] | .name`)
   - `.key` nested access — dot-path like `.data.name` (currently only handles one level)
   - `keys` — return object keys as array
   - `values` — return object values as array (currently mapped to `.[]` but should be distinct)
   
   The fallback doesn't need a full jq interpreter — only the specific filters actually used in the callsites below. Catalog each filter before implementing. Per CONTEXT.md: "JavaScript fallback mirrors jq logic with full feature parity for every filter used."

   IMPORTANT: Update FILTER_PRESETS to fix the current incorrect mappings (e.g., `keys: '.[]'` should be `keys: 'keys'`; `values: '.[]'` should be `values: 'values'`).

**tools.js integration (src/commands/tools.js):**
2. In `cmdDetectTools()` (the function that outputs detect:tools JSON), add jq transformation as an optimization:
   - After building the tool status array, use `transformJson(statusArray, '[.[] | select(.available == true) | .name]', { compact: true })` to extract available tool names as a summary line.
   - Add this summary to the output (e.g., `"available_summary": ["ripgrep", "fd", "jq"]`).
   - The existing JSON output format remains unchanged — this is an additive enhancement.
   - If jq fails or falls back, use the JS equivalent: `statusArray.filter(t => t.available).map(t => t.name)`.

**codebase-intel.js integration (src/lib/codebase-intel.js):**
3. In `readIntel()` around line 372-383, when reading large intel JSON files, use jq for field extraction:
   - For large files (>100KB), use `transformJson(content, '.files | keys | length', { raw: true })` to quickly get file count without parsing the entire JSON into a JS object.
   - This is an optimization for large codebases where JSON.parse of multi-MB intel files is slow.
   - Fallback: existing JSON.parse behavior (no change needed — the wrapper handles it).

   - Fallback handles the case transparently.
