**Architecture:** Single-file CLI with modular tool detection layer, graceful fallback patterns, and integration at 3 levels — detection, invocation, and agent routing.

**Current state:**
- Tool detection infrastructure exists (`src/lib/cli-tools/detector.js`) with 5-minute caching
- 6 tools already wrapped: ripgrep, fd, jq, yq, bat, gh
- Fallback pattern implemented (`src/lib/cli-tools/fallback.js`)
- Research degradation tier calculated by tool availability (v12.0)
- Only `util:tools` command currently routes to tool detection
- Decision functions exist for workflow routing but not tool selection

**Integration locations:**
- `src/lib/cli-tools/` (detection, invocation, fallback) — **11 files, ~500 lines**
- `src/commands/` (command-specific tool usage) — **18 command modules**
- `src/lib/decision-rules.js` (decision functions) — **708 lines, partial tool routing**
- `src/router.js` (namespace routing, lazy loading) — **1360 lines**
- Agent context via `init:*` commands in `src/commands/init.js`

**Key patterns:** 
- **Graceful degradation:** 4-tier fallback (CLI → Node.js fallback → guidance → skip)
- **Lazy module loading:** Tools loaded on first use (not at startup)
- **5-minute detection caching:** Tool availability checked once per 5 min per process
- **SQL-backed cache:** Persistent caching via SQLite (v12.0 foundation)
- **Manifest-driven context:** Agent context includes tool availability

**Anti-patterns to avoid:**
- Detecting tool twice in same invocation (defeats caching)
- Hardcoding tool paths (breaks portability)
- Tool missing = error (violates graceful degradation)
- Tool invocation without fallback (crashes workflow)
- Agent unaware of available tools (suboptimal decisions)

**Integration priority:**
1. **Expand decision functions** for tool selection (degradation tier exists; tool-based routing missing)
2. **Wire tool detection into more command modules** — codebase.js, verify.js, features.js currently don't use tools
3. **Create shared tool invocation pattern** to avoid copy-paste in commands
4. **Extend agent context** with available/unavailable tools + capability levels
5. **Add tool-specific error recovery** for network/permission issues

### Data Flow: Tool Detection to Command Execution

```
User command: util:tools status
      ↓
Router dispatches to cmdToolsStatus() in src/commands/tools.js
      ↓
Call getToolStatus() from detector.js
      ↓
For each tool in TOOLS registry:
  - Check toolCache[toolName] for valid entry (timestamp < 5min ago?)
  - If cache miss or expired:
      └→ execFileSync('toolname', ['--version'])
      └→ Parse version output
      └→ Store in toolCache with timestamp
  - Return { available, path, version }
      ↓
Separate into available/unavailable lists
      ↓
Output formatted (TTY: pretty table) or JSON (piped: raw data)
```

### Graceful Degradation: 4-Tier Fallback Chain

When a command needs a tool, cascading fallback ensures it always works:

```
┌─────────────────────────────────────────────────┐
│ Tier 1: CLI Tool Available                      │
│ • Detect available via detector.js cache       │
│ • Execute via execFileSync, parse output       │
│ • Best performance (10-100x faster)            │
└──────────────┬──────────────────────────────────┘
               │ Tool missing
               ↓
┌─────────────────────────────────────────────────┐
│ Tier 2: Node.js Fallback Available              │
│ • withToolFallback() wrapper executes fallback  │
│ • Example: rg → regex search in Node.js         │
│ • Acceptable performance                       │
└──────────────┬──────────────────────────────────┘
               │ Fallback also fails
               ↓
┌─────────────────────────────────────────────────┐
│ Tier 3: Partial Capability + Guidance           │
│ • Return null result + install guidance string │
│ • Example: "Install ripgrep: brew install rg" │
│ • Workflow continues (non-blocking)            │
└──────────────┬──────────────────────────────────┘
               │ No fallback possible
               ↓
┌─────────────────────────────────────────────────┐
│ Tier 4: Feature Skipped                         │
│ • Return empty result, continue execution      │
│ • Example: Skip ripgrep search, continue with  │
│   next task in plan                            │
│ • Workflow proceeds (no dead-end)              │
└─────────────────────────────────────────────────┘
```

**Pattern in code (fallback.js):**
```javascript
function withToolFallback(toolName, cliFn, nodeJsFallback) {
  const toolResult = detectTool(toolName);
  
  if (toolResult.available) {
    try {
      return {
        success: true,
        usedFallback: false,
        result: cliFn()  // Tier 1: Use CLI
      };
    } catch (error) {
      return { success: false, usedFallback: false, error: error.message };
    }
  }
  
  // Tool missing → try Node.js
  try {
    const guidance = getInstallGuidance(toolName);
    return {
      success: true,
      usedFallback: true,
      guidance: { installCommand: '...' },
      result: nodeJsFallback()  // Tier 2: Use fallback
    };
  } catch (error) {
    const guidance = getInstallGuidance(toolName);
    return {
      success: false,
      usedFallback: true,
      guidance,
      error: error.message  // Tier 3/4: Return guidance
    };
  }
}
```

### Tool Invocation Patterns

When a command needs to invoke a tool, three styles are used (pick one per use case):

**Style 1: Simple availability check (lowest coupling)**
```javascript
const { isRipgrepAvailable } = require('../lib/cli-tools/ripgrep');

if (isRipgrepAvailable()) {
  const results = searchRipgrep(pattern);
} else {
  // Skip feature or use fallback
}
```

**Style 2: Fallback wrapper (recommended for important operations)**
```javascript
const { withToolFallback } = require('../lib/cli-tools/fallback');

const result = withToolFallback(
  'ripgrep',
  () => execFileSync('rg', ['pattern', path]),
  () => fallbackGrep(pattern, path)  // Node.js fallback
);

if (result.usedFallback) {
  console.log('Note: Using slower search. ' + result.guidance.installCommand);
}
```

**Style 3: Index convenience pipeline (for chained operations)**
```javascript
const { searchFiles, searchAndTransform } = require('../lib/cli-tools');

const results = searchFiles(pattern, '*.js', {
  contentOptions: { type: 'match' },
  fileOptions: { type: 'f' }
});
// Internally: fd → ripgrep → parse JSON, with automatic fallback
```

```javascript
/**
 * Determine if fast file discovery should be used (fd vs Node.js).
 */
function resolveFileDiscoveryMode(state) {
  const {
    fd_available = false,
    file_count = 0,
    performance_critical = false
  } = state || {};

  // If fd available and either high file count or critical path
  if (fd_available && (file_count > 1000 || performance_critical)) {
    return { value: 'fd', confidence: 'HIGH' };
  }
  
  // If fd available but low file count, still use fd (always better)
  if (fd_available) {
    return { value: 'fd', confidence: 'MEDIUM' };
  }
  
  // Fallback to Node.js (reliable)
  return { value: 'node', confidence: 'HIGH' };
}

/**
 * Determine if pattern search should use ripgrep or Node.js regex.
 */
function resolveSearchMode(state) {
  const {
    ripgrep_available = false,
    pattern_type = 'simple',
    search_size = 'small'
  } = state || {};

  // If ripgrep available and complex operation
  if (ripgrep_available && (search_size === 'large' || pattern_type === 'complex')) {
    return { value: 'ripgrep', confidence: 'HIGH' };
  }
  
  // If ripgrep available, use it
  if (ripgrep_available) {
    return { value: 'ripgrep', confidence: 'MEDIUM' };
  }
  
  // Fall back to Node.js regex
  return { value: 'regex', confidence: 'HIGH' };
}

/**
 * Determine if JSON transformation should use jq or Node.js.
 */
function resolveJsonTransformMode(state) {
  const {
    jq_available = false,
    data_size = 'small'
  } = state || {};

  // If jq available and large payload
  if (jq_available && data_size === 'large') {
    return { value: 'jq', confidence: 'HIGH' };
  }
  
  if (jq_available) {
    return { value: 'jq', confidence: 'MEDIUM' };
  }
  
  // Node.js JSON.parse + manipulation
  return { value: 'node', confidence: 'HIGH' };
}

/**
 * Determine agent's overall capability level based on available tools.
 */
function resolveAgentCapabilityLevel(state) {
  const { tool_availability = {} } = state || {};
  const availableCount = Object.values(tool_availability).filter(Boolean).length;

  if (availableCount >= 5) {
    return {
      value: 'full',
      confidence: 'HIGH',
      tools: ['ripgrep', 'fd', 'jq', 'yq', 'bat', 'gh']
    };
  }
  if (availableCount >= 2) {
    return {
      value: 'enhanced',
      confidence: 'HIGH',
      tools: Object.keys(tool_availability).filter(t => tool_availability[t])
    };
  }
  return {
    value: 'standard',
    confidence: 'HIGH',
    tools: []
  };
}
```

**How consumed:**
- In workflows via `bgsd-context` enricher
- In command modules for branching logic
- In agent prompts for optimizing task decomposition

Before (existing, slow):
```javascript
// src/commands/codebase.js
function analyzeCodebase(cwd) {
  const files = findFilesSync(cwd); // fs.readdirSync recursively
  // Takes 2+ seconds for 10K files
  const imports = parseImports(files);
  return { files, imports };

```

After (with fallback):
```javascript
const { withToolFallback, getInstallGuidance } = require('../lib/cli-tools/fallback');

function analyzeCodebase(cwd) {
  // Try ripgrep for fast discovery, fall back to fs.readdirSync
  const result = withToolFallback(
    'fd',
    () => {
      // Fast path: fd finds files in ~100ms
      const fdResult = execFileSync('fd', ['--type', 'f'], { cwd });
      return fdResult.toString().split('\n').filter(Boolean);
    },
    () => {
      // Fallback: Node.js fs (slower but works)
      return findFilesSync(cwd);
    }
  );

  if (result.usedFallback) {
    debugLog('codebase', `Using Node.js file discovery. ${result.guidance?.installCommand || ''}`);
  }

  const files = result.result;
  const imports = parseImports(files);
  return { files, imports };
}
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Detecting Tool Twice

**What people do:**
```javascript
// BAD: Tool detection called twice
if (isRipgrepAvailable()) {          // Detection #1
  const result = searchRipgrep(pattern);
}
// Later in different function:
if (isRipgrepAvailable()) {          // Detection #2 (cache miss if 5min elapsed)
  processResult(result);
}
```

**Why it's wrong:** Wastes 5-minute cache efficiency, runs tool detection N times per invocation

**Do this instead:**
```javascript
// GOOD: Detect once, use result
const { withToolFallback } = require('../lib/cli-tools/fallback');
const result = withToolFallback(
  'ripgrep',
  () => searchRipgrep(pattern),
  () => fallbackGrep(pattern)
);
// result.success, result.result, result.usedFallback available for branching
```

---

### Anti-Pattern 2: Hardcoding Tool Paths

**What people do:**
```javascript
// BAD: Hardcoded path
const rg = '/usr/bin/rg';           // Not in standard location on macOS
const output = execFileSync(rg, ['pattern', path]);
```

**Why it's wrong:** Breaks on different systems, ignores user's $PATH, can't use aliases

**Do this instead:**
```javascript
// GOOD: Let detectTool find it, use plain command name
const { detectTool } = require('./detector');
const result = detectTool('ripgrep');
if (result.available) {
  // execFileSync('rg', ...) uses $PATH automatically
  const output = execFileSync('rg', ['pattern', path]);
}
```

---

### Anti-Pattern 3: Tool Missing = Error

**What people do:**
```javascript
// BAD: Throws error if tool missing
try {
  const output = execFileSync('rg', ['pattern', path]);
} catch (e) {
  throw new Error('ripgrep required'); // Breaks workflow for all users without tool
}
```

**Why it's wrong:** Users without the tool can't use bGSD; violates graceful degradation principle

**Do this instead:**
```javascript
// GOOD: Missing tool = fallback, warning to stderr
const result = withToolFallback(
  'ripgrep',
  () => execFileSync('rg', ['pattern', path]),
  () => fallbackGrep(pattern, path)  // Node.js version works without tool
);

if (result.usedFallback && result.guidance) {
  status(`Using slower Node.js search. Install ripgrep: ${result.guidance.installCommand}`);
}
```

---

### Anti-Pattern 4: Tool Invocation Without Context

**What people do:**
```javascript
// BAD: Invokes tool without checking availability first
const { spawnSync } = require('child_process');
const result = spawnSync('rg', ['pattern'], { cwd });
if (result.error) {
  // Fails with "ENOENT: command not found"
  console.error('ripgrep not found');
}
```

**Why it's wrong:** Can't provide install guidance, can't fall back automatically, creates confusing error messages

**Do this instead:**
```javascript
// GOOD: Check availability first
const { searchRipgrep, isRipgrepAvailable } = require('../lib/cli-tools/ripgrep');

if (!isRipgrepAvailable()) {
  const guidance = getInstallGuidance('ripgrep');
  status(`ripgrep not available. Install: ${guidance.installCommand}`);
  return fallbackSearch(...);  // Use fallback seamlessly
}

const results = searchRipgrep(pattern);
```

---

### Anti-Pattern 5: Agent Unaware of Available Tools

**What people do:**
```javascript
// BAD: Agent doesn't know tool status
const context = gatherContext(cwd);
// { state, roadmap, codebase } — no tool_availability field
// Agent must assume tools missing or guess availability
```

**Why it's wrong:** Agent can't optimize task decomposition, creates CPU-bound tasks when ripgrep available, suboptimal plans

**Do this instead:**
```javascript
// GOOD: Include tool availability in context
const context = {
  ...existing,
  
  // Agent now sees which tools are available
  tool_availability: {
    ripgrep: isRipgrepAvailable(),
    fd: isFdAvailable(),
    jq: isJqAvailable(),
    // ... others ...
  },
  
  // Agent knows overall capability level
  agent_capability: resolveAgentCapabilityLevel({
    tool_availability: { ripgrep: true, fd: true, jq: false, ... }
  })
  
  // Agent sees install guidance if tool missing
};

// Agent can now route tasks optimally:
// - With ripgrep: decompose search tasks (ripgrep is fast)
// - Without ripgrep: combine search tasks (Node.js regex is slow)
```

## Scaling Considerations

| Developer Count | Tool Integration Strategy | Notes |
|---|---|---|
| **Solo dev (1)** | Graceful degradation works fine. Tools optional. bGSD functions without any CLI tools via pure Node.js fallbacks. | Start without tools, add if codebase grows. |
| **Small team (2-5)** | ripgrep + fd recommended (10-100x file discovery speedup). Payoff: faster `init:plan-phase` and `verify:*` operations. | Install: `brew install ripgrep fd` |
| **Medium team (5-20)** | ripgrep + fd essential. jq for JSON test results. gh for GitHub integration. yq for config parsing. | All 6 tools recommended. bGSD detects, runs optimized paths. |
| **Large team (20+)** | All 6 tools recommended. Tool detection caching critical (5min TTL saves 100+ tool checks/session). SQLite cache persistence (v12.0) persists detection across sessions. | Tools accelerate shared codebase analysis, agent context generation. |

### Scaling Bottlenecks

**First bottleneck: File discovery on large codebases (1K+ files)**
- **What breaks:** Codebase-intel scanning slow if using fs.readdirSync (~500ms for 5K files)
- **When:** Phase where `codebase.js` analyzes large monorepo
- **How to fix:** Wire fd into codebase.js discovery (~50ms for 5K files with fd)
- **Impact:** 10x speedup on init operations

**Second bottleneck: Large JSON parsing (MCP profiles, test results)**
- **What breaks:** Node.js JSON manipulation slow on >50KB payloads
- **When:** Phase where agents process large test result files from CI
- **How to fix:** Wire jq into features.js for filtering/transformation
- **Impact:** 2-5x speedup on parsing operations

**Third bottleneck: Search in large codebase (10K+ files, 1M+ lines)**
- **What breaks:** Regex scanning in Node.js takes seconds
- **When:** Phase where verify.js searches for assertions, test files
- **How to fix:** Wire ripgrep into verify.js
- **Impact:** 100x speedup on search operations

## Decision Functions Cheat Sheet

When building features that need tool routing, add these to `decision-rules.js`:

```javascript
/**
 * Should fast file discovery (fd) be used?
 * Route: fd available + (many files OR performance critical) → 'fd'
 * Route: fd available + low file count → 'fd' (still better)
 * Route: fd missing → 'node' (fallback)
 */
function resolveFileDiscoveryMode(state) {
  const { fd_available = false, file_count = 0, performance_critical = false } = state || {};
  
  if (fd_available && (file_count > 1000 || performance_critical)) {
    return { value: 'fd', confidence: 'HIGH' };
  }
  if (fd_available) {
    return { value: 'fd', confidence: 'MEDIUM' };
  }
  return { value: 'node', confidence: 'HIGH' };
}

/**
 * Should pattern search use ripgrep?
 * Route: ripgrep available + (large search OR complex pattern) → 'ripgrep'
 * Route: ripgrep available + simple search → 'ripgrep' (still better)
 * Route: ripgrep missing → 'regex' (Node.js regex fallback)
 */
function resolveSearchMode(state) {
  const { ripgrep_available = false, search_size = 'small', pattern_type = 'simple' } = state || {};
  
  if (ripgrep_available && (search_size === 'large' || pattern_type === 'complex')) {
    return { value: 'ripgrep', confidence: 'HIGH' };
  }
  if (ripgrep_available) {
    return { value: 'ripgrep', confidence: 'MEDIUM' };
  }
  return { value: 'regex', confidence: 'HIGH' };
}

/**
 * What's the agent's overall capability level?
 * Based on count of available tools.
 */
function resolveAgentCapabilityLevel(state) {
  const { tool_availability = {} } = state || {};
  const count = Object.values(tool_availability).filter(Boolean).length;
  
  if (count >= 5) {
    return {
      value: 'full',
      confidence: 'HIGH',
      description: 'All tools available: ripgrep, fd, jq, yq, bat, gh'
    };
  }
  if (count >= 2) {
    return {
      value: 'enhanced',
      confidence: 'HIGH',
      description: `${count} tools available for optimization`
    };
  }
  return {
    value: 'standard',
    confidence: 'HIGH',
    description: 'No CLI tools; using Node.js fallbacks'
  };
}
```

Then consume in workflows/agents:
```markdown
**Agent Capability:** {{ agent_capability.value }}  
{{ agent_capability.description }}

_The executor has **{{ agent_capability.value }}** capability._
- **full**: Use all optimization tricks (ripgrep + fd + jq + yq)
- **enhanced**: Use fast discovery + search, JavaScript for transform
- **standard**: Use Node.js fallbacks, be patient with large codebases
```
